# P0 级任务整理与 Codex Agent 开发提示词

## 一、P0 级任务定位

P0 不是继续堆新功能，而是优先补齐那些会直接影响**稳定交付、稳定演示、稳定维护**的基础能力。它们决定系统能不能作为一个可持续开发、可答辩、可验收的企业级项目继续推进。包括：

1. 数据库迁移执行与启动前检查机制
2. 最小 CI / 自动化验证流水线
3. 统一日志、异常提示与错误回退治理
4. 统一接口清单、模块边界、数据口径与交付文档持续维护机制
5. 管理员与教师权限边界进一步收口
6. 演示数据、测试数据与初始化脚本标准化
7. 主链路回归脚本持续扩充

## ~~三、P0 任务 1：启动前检查与迁移治理~~

### 任务拆解

目标是防止再次出现“代码已经读取新字段，但数据库还没迁移”这类问题。后续方向包括：启动前检查迁移是否已执行、检查关键依赖是否安装、检查 MySQL 连接与必要表结构、明确迁移执行顺序、把迁移检查纳入演示脚本。

### 可直接复制给 Codex 的提示词

```text
你现在要完成的任务是：启动前检查与迁移治理。

【任务目标】
在当前系统多轮增量开发的基础上，建立最小可用的启动前检查与迁移治理机制，重点防止“代码已读新字段，但数据库未迁移”之类的问题再次出现。

【必须完成】
- 扫描当前后端启动方式、manage.py、配置文件、README、部署说明和已有脚本。
- 梳理当前数据库迁移执行方式和启动依赖。
- 增加启动前检查机制，至少覆盖以下方面：
  - 检查迁移是否已执行
  - 检查关键依赖是否安装
  - 检查 MySQL 连接是否可用
  - 检查必要表结构是否存在
- 增加或完善迁移执行顺序说明。
- 在演示脚本或启动文档中加入迁移检查步骤。
- 若仓库已有类似脚本，优先增强而不是重复新建一套并行方案。

【实施原则】
- 优先使用现有 Python / Django 管理命令、启动脚本、文档体系。

【特别禁止】
- 不要引入复杂部署平台。
- 不要重写整个启动流程。
- 不要做与当前仓库技术栈不一致的方案。

请直接在仓库中完成代码、脚本、文档和验证说明修改。
```

## ~~四、P0 任务 2：最小 CI 与自动化验证流水线~~

### 任务拆解

目标是让关键验证不再依赖人工记忆。后续方向包括接入最小 CI，自动执行后端 `manage.py check`、后端关键测试、前端 `verify:baseline`、统一主链路回归脚本，并为失败结果提供最小可读日志。

```text
你现在要完成的任务是：最小 CI 与自动化验证流水线。

【任务目标】
为当前系统建立最小可用的 CI / 自动化验证流水线，让关键验证不再依赖人工记忆，形成企业级持续开发的最小防回归装置。

【必须完成】
- 扫描当前仓库中的前端命令、后端命令、测试脚本、基线验证脚本、CI 配置文件和文档。
- 识别真实存在且可运行的验证命令。
- 建立最小 CI 或自动化验证流水线，至少自动执行：
  - 后端 manage.py check
  - 后端关键测试
  - 前端 verify:baseline 或等价基线脚本
  - 统一主链路回归脚本
- 为失败结果提供最小可读日志或输出说明。

【实施原则】

【特别禁止】
- 不要凭空假设 npm / pnpm / yarn / pytest / unittest，必须以仓库真实配置为准。

请直接完成 CI / 脚本 / 文档 / 验证说明修改。
```

这个方向的目标是让推荐从当前“规则增强型推荐 + 推荐解释维度”继续走向“有反馈、有历史、有摘要、可持续优化”的模块。规划文档明确的方向包括：收藏、点击、关注、申请意向等反馈采集，反馈对推荐排序的影响，反馈可视化摘要，管理员分析教师对推荐响应情况；边界文档同时明确：当前推荐仍是规则增强路线，后续要先补用户反馈信号，再评估是否进入更强模型化推荐。

### 可直接复制给 Codex 的提示词

```text
你现在要完成的任务是：推荐反馈闭环与策略持续优化。

【任务目标】
在当前规则增强型推荐已经可展示、可解释的基础上，建立最小可用的反馈闭环，让推荐逐步具备反馈采集、反馈摘要、响应分析和后续策略优化基础。

【必须完成】
- 扫描当前推荐页面、推荐接口、推荐解释字段、收藏相关逻辑、管理员推荐视图、教师推荐视图和相关类型定义。
- 在以下反馈信号中至少落地一个最小闭环，能真实采集、保存并展示：
  - 收藏
  - 点击
  - 关注
  - 申请意向
- 设计并实现最小可用的反馈记录结构。
- 在推荐页面或相关摘要区展示基础反馈结果。
- 在管理员视角增加教师对推荐响应情况的最小分析摘要。
- 如条件允许，增加推荐历史或反馈历史中的至少一部分能力。
- 为后续“反馈影响排序”预留清晰的数据结构或接口边界，但当前阶段不要求直接做复杂排序模型。
- 为关键反馈链路补最小测试或验证步骤。

【必须保留】
- 当前推荐仍是规则增强路线。
- 当前阶段先做反馈闭环，再评估更强模型化推荐。
- 不能因为新增反馈能力而破坏现有推荐结果主链路。

【实施原则】
- 优先落地最小可闭环、可验证、可积累的反馈能力。
- 优先复用现有推荐结果页、管理员摘要页和现有推荐解释框架。
- 反馈展示要有教师视角和管理员视角，但权限边界必须清晰。
- 若需要新增表或字段，必须最小化设计，并补迁移与兼容说明。

【验收要求】
- 推荐结果不再只是“给结果”，而是形成最小反馈闭环。
- 至少有一类反馈信号可持久化和可展示。
- 管理员能够看到基础响应摘要。
- 文档清楚说明当前反馈闭环能力、数据结构边界和后续优化路线。

【特别禁止】
- 不要在反馈数据很少时直接上复杂学习排序或机器学习推荐。
- 不要为了反馈闭环大规模重构推荐底层。
- 不要把未来高级推荐优化能力写成当前已上线能力。

请直接完成代码、类型、测试、文档和验证说明。
```

## 六、P2 任务 4：智能问答知识来源治理与模板扩展

### 任务拆解

这个方向的目标是让当前基于真实系统数据的模板化问答继续扩展，但仍然保持受控、可解释、可回退。规划文档明确的方向包括：更多模板化问题、更多来源卡片与边界说明、更强的推荐/画像/看板联动问答、问答失败回退策略细化，以及未来 RAG 前置方案设计；边界文档同时明确：当前问答不是通用知识平台，也没有 RAG、向量检索或外部知识库，因此当前必须坚持系统内真实数据问答，不包装成完整知识平台。

### 可直接复制给 Codex 的提示词

```text
你现在要完成的任务是：智能问答知识来源治理与模板扩展。

【任务目标】
在当前智能问答已支持基于真实系统数据的模板化问答、来源说明、边界提示和失败回退的基础上，继续增强问答的覆盖范围、来源治理能力和跨模块解释能力，但仍保持受控、可解释、可回退。

【必须完成】
- 扫描当前问答页面、问答模板、问答接口、数据来源拼装逻辑、来源卡片、边界说明、失败回退逻辑和相关类型定义。
- 扩展更多模板化问题，优先覆盖以下方向中的一部分：
  - 画像相关问题
  - 推荐相关问题
  - 看板相关问题
  - 成果与画像、成果与推荐联动问题
- 增强来源卡片展示，让答案更清楚地说明“数据来自哪些模块、哪些页面可验证”。
- 增强边界说明与失败回退策略，至少覆盖：
  - 无数据
  - 数据不足
  - 权限不足
  - 图谱降级 / 推荐降级 / 画像快照缺失等场景
- 支持问答结果跳转到对应页面或证据区中的至少一部分能力。
- 在文档中明确当前问答仍然是系统内真实数据驱动的模板化问答，不是通用知识平台。
- 为关键问答链路补最小测试或验证步骤。

【必须保留】
- 当前仍坚持系统内真实数据问答。
- 当前不引入 RAG、向量检索、外部知识库。
- 不把模板化解释包装成“无边界的智能回答”。

【实施原则】
- 优先增强来源治理、模板扩展、失败回退和页面证据联动。
- 优先复用现有问答模板机制和来源说明结构。
- 不能把推断包装成确定事实，所有回答都要与系统内数据口径一致。
- 对无法回答的问题，应明确说明原因和可跳转验证路径。

【验收要求】
- 模板化问题覆盖范围更广。
- 来源卡片和边界说明更清晰。
- 问答失败时用户更容易理解原因和后续操作。
- 至少有一部分问答结果可以回跳到页面证据区。
- 文档说明当前问答边界与未来可扩展方向。

【特别禁止】
- 不要在知识来源不清时直接做重型 RAG。
- 不要把系统外部知识包装成系统内事实。
- 不要为了问答增强而重写整个问答主链路。

请直接完成代码、类型、测试、文档和验证说明。
```

## 七、P2 任务 5：跨模块联动增强

### 任务拆解

这个方向的目标是把当前已经具备雏形的资料、成果、画像、图谱、推荐、问答之间的联动，继续强化成真正的系统闭环。规划文档明确的方向包括：从成果跳转到画像对应维度、从画像跳转到推荐原因、从推荐跳转到支撑成果、从看板跳转到画像/推荐/问答、从问答回跳到对应页面证据区。边界文档也明确指出，企业级完整版本不能是孤立模块拼接，而必须形成联动系统。

### 可直接复制给 Codex 的提示词

```text
你现在要完成的任务是：跨模块联动增强。

【任务目标】
在当前系统已经具备资料、成果、画像、图谱、推荐、问答之间部分联动雏形的基础上，继续增强跨模块跳转、证据链回跳和解释链闭环，让系统更像“联动平台”而不是“孤立模块拼接”。

【必须完成】
- 扫描当前成果、画像、图谱、推荐、问答、看板相关页面、路由、详情页、摘要卡片、跳转入口和相关类型定义。
- 梳理当前已存在的联动能力和断点。
- 优先完成以下方向中的多数项：
  - 从成果跳转到画像对应维度
  - 从画像跳转到推荐原因或支撑成果
  - 从推荐跳转到支撑成果、画像维度或指南详情
  - 从看板跳转到画像、推荐或问答
  - 从问答回跳到对应页面证据区
- 为跨模块联动增加清晰的跳转参数、定位逻辑和证据区说明。
- 对教师视角和管理员视角分别检查权限边界，避免跨模块联动造成越权。
- 在文档中明确当前已经打通的联动链路和仍未完全打通的边界。
- 为关键联动链路补最小测试或验证步骤。

【必须保留】
- 联动必须建立在真实数据与真实页面证据基础上。
- 所有联动仍需接入现有严格守卫。

- 所有跳转必须可解释、可回退、可定位。
- 联动要尽量同时服务教师与管理员两个视角，但不能混淆权限边界。

【验收要求】
- 系统模块之间的跳转链路更完整。
- 用户能从一个结果追到其对应证据页面或支撑模块。
- 至少有多条高价值跨模块联动链路可稳定演示。
- 文档说明当前联动闭环范围和下一步扩展方向。

【特别禁止】
- 不要做“看起来能跳，实际没有证据区”的假联动。
- 不要为了联动大规模重构所有页面。
- 不要忽略教师 / 管理员视角下的权限边界。
```

## 八、建议执行顺序

如果 P0、P1 已经基本完成，P2 最稳的推进顺序建议是：

1. 跨模块联动增强
2. 智能问答知识来源治理与模板扩展
3. 推荐反馈闭环与策略持续优化
4. 画像快照与阶段对比
5. 学术图谱与图分析增强

这个顺序的逻辑是：

- 先把 **模块之间的证据链和跳转闭环** 打通，系统整体感会马上增强。
- 再做 **问答来源治理与模板扩展**，能更快把解释能力做出来。
- 再做 **推荐反馈闭环**，为后续推荐优化打基础。
- **画像快照** 和 **图谱路径分析** 更容易涉及口径设计、数据结构和降级兼容，所以放后面更稳。

这个排序也和你规划文档里“第三阶段：分析与智能深化”的节奏相符，属于先做闭环与解释，再做更深分析能力的路线。
