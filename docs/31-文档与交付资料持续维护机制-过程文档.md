# 31-文档与交付资料持续维护机制-过程文档
最后更新时间：2026-03-21

## 1. 任务目标

本轮目标不是继续堆新的过程记录，而是在现有 README、统一现状文档、演示验收文档和遗留问题记录的基础上，补一套持续可维护的文档机制，避免后续再次回到“只能看过程，看不清现状”的状态。

## 2. 扫描结果与主要缺口

本轮扫描了：

- `README.md`
- `docs/系统文档索引与交付总览.md`
- `docs/系统接口清单-统一版.md`
- `docs/模块边界与系统最终形态说明-统一版.md`
- `docs/数据口径与风险记录-统一版.md`
- `docs/最终答辩演示步骤-第三轮.md`
- `docs/最终验收清单-第三轮.md`
- `docs/遗留问题记录.md`

确认到的主要缺口：

- 已有统一现状文档，但缺少统一的维护规则说明
- 文档索引还没有把“看现状先看什么”表达得足够清楚
- 若每轮开发后不主动同步现状文档，仍容易退回到“只看过程文档”的状态
- 部分统一文档里已有过时表述，例如工程化章节仍写着“没有正式 CI 工作流”

## 3. 本轮处理

### 3.1 新增统一维护机制文档

新增：

- `docs/文档维护规则与交付资料机制-统一版.md`

核心内容包括：

- 文档分层
- 统一现状文档职责
- 每轮功能变更后的强制同步项
- 推荐的更新顺序
- 已完成能力与预留边界的写法规则
- 论文写作稳定口径

### 3.2 收口现有入口文档

更新：

- `README.md`
- `docs/系统文档索引与交付总览.md`

使它们都能明确指向：

- 统一现状文档入口
- 文档维护规则
- 过程文档和现状文档的职责边界

### 3.3 同步统一现状文档

更新：

- `docs/系统接口清单-统一版.md`

重点修正了：

- 模块边界文档中“没有正式 CI 工作流”的过时表述

### 3.4 同步演示、验收与遗留问题文档

更新：

目的：

- 让演示、验收和遗留问题记录都能回到统一现状文档体系
- 明确后续每轮变更需要同步更新的资料入口

## 4. 关键设计决策

- 没有推翻现有文档体系，而是在原有统一文档之上补“持续维护规则”
- 没有简单重复新写一套接口、边界和数据文档，而是以当前已有统一文档为主入口
- 明确区分：
  - 现状文档
  - 演示验收文档
  - 多轮过程文档
- 明确要求后续每轮变更都回写接口、模块边界、数据口径、风险和演示主链路

## 5. 验证说明

本轮主要是文档整理与现状收口，不涉及业务代码逻辑变更。

建议至少完成以下验证：

- 确认 `README.md` 已新增文档持续维护机制入口
- 确认 `docs/系统文档索引与交付总览.md` 已把“看现状”与“看过程”区分开
- 确认新文档 `docs/文档维护规则与交付资料机制-统一版.md` 已被索引和 README 引用
- 确认模块边界文档已修正过时的 CI 表述

## 6. 当前边界

- 本轮没有重写旧过程文档正文，只是在保留历史记录的前提下增强现状入口和维护机制
- 本轮没有生成自动化文档站点或 OpenAPI 文档平台
- 本轮没有改动业务逻辑，只修正文档结构和文档口径

## 7. 后续建议

后续每轮开发建议遵循同一规则：

1. 代码和验证完成后，先回写统一现状文档
2. 再更新演示和验收文档
3. 最后新增当轮过程文档

这样才能让仓库长期保持“既能看过程，也能快速看现状”的可维护状态。
