# 学术图谱与图分析增强-过程文档

## 1. 本次任务目标

本次任务在现有图谱展示已可用、Neo4j 可选、MySQL 回退已具备的基础上，增强图谱页的分析价值与解释性，使图谱不仅能“看关系”，也能用于轻量分析与讲述。

本次坚持以下约束：

- Neo4j 不能成为论文录入和画像展示的强依赖
- 不重做图谱架构
- 不引入复杂图挖掘全套能力
- 不让高级图分析影响基础图谱主链路

## 2. 仓库扫描结论

### 2.1 后端现状

图谱主接口位于：

- `backend/graph_engine/views.py`

当前已有能力：

- 优先尝试 Neo4j 实时读取
- Neo4j 不可用时自动回退到 MySQL 关系数据
- 返回节点、边、基础 `meta` 和简单 `analysis`

当前已有分析结果偏轻：

- 最活跃合作者
- 高频研究主题
- 节点与关系数量

### 2.2 前端现状

图谱页组件位于：

- `frontend/src/views/AcademicGraph.vue`

当前已有展示：

- 力导向图谱
- 节点类型显隐
- 节点/边详情
- 数据来源说明
- 轻量热点卡

不足点：

- 侧栏分析信息仍偏少
- 图谱降级提示还不够明确
- 缺少对计算口径与数据来源边界的清晰解释

## 3. 本次实施方案

### 3.1 后端增强策略

不改图谱主接口路径，仅增强返回数据结构，并将图分析聚合从视图中拆出到独立模块：

- 新增 `backend/graph_engine/analysis.py`
- `views.py` 中统一调用分析构造器
- `meta` 中补充：
  - `analysis_level`
  - `calculation_note`
  - `fallback_tip`
- `analysis` 中补充：
  - `collaboration_overview`
  - `collaborator_type_breakdown`
  - `theme_hotspots`
  - `analysis_method_note`

### 3.2 本次新增的轻量图分析结果

本次落地了 3 类轻量图分析结果：

1. 合作网络概览
   - 合作者数量
   - 合作边数量
   - 篇均合作者数量
   - 最强合作对象

2. 合作者类型分析
   - 校内合作数量
   - 校外合作数量
   - 校内外占比
   - 计算说明

3. 研究主题热点分析
   - 高频关键词
   - Top3 关键词集中度
   - 近 3 年关键词热点变化
   - 当前主题聚焦标签

### 3.3 Neo4j 与 MySQL 回退边界

当前边界保持不变：

- Neo4j 可用时，图谱节点与边优先来自图数据库
- Neo4j 不可用时，回退到 MySQL 关系数据
- 图分析结果本次统一以现有关系数据和成果数据做轻量统计

这样做的原因：

- 保证图谱展示和图分析都不依赖 Neo4j 才能工作
- 避免“有分析就必须依赖图数据库”的强耦合
- 保持录入链路和画像主链路稳定

## 4. 关键设计决策与权衡

### 4.1 为什么图分析结果统一基于关系数据统计

虽然 Neo4j 图谱已经可选接入，但当前系统的稳定主链路仍是关系数据。

如果把新增图分析只建立在 Neo4j 查询上，会导致：

- Neo4j 异常时分析结果整体消失
- 图谱页体验退化明显
- 违背“Neo4j 不能成为强依赖”的原则

因此本次图分析统一按关系数据与已有成果数据做轻量统计，Neo4j 主要继续承担图谱展示的实时读取角色。

### 4.2 为什么不做复杂路径分析和合作圈层分析

当前系统尚未建立稳定的：

- 历史图快照
- 节点权重治理
- 社区划分口径
- 路径搜索与解释机制

若在本次强行加入复杂图分析，容易出现“有结果但难解释”的问题，也会显著增加维护成本。

因此本次只补“可讲述、可演示、可解释”的轻量图分析结果。

## 6. 当前图分析能力级别说明

当前级别：

- `lightweight-analysis`

当前适合用于：

- 展示合作网络概览
- 展示校内外合作结构
- 展示研究主题热点和近年热点
- 作为教师画像页中的关系分析补充

当前不应被表述为：

- 完整图挖掘平台
- 社区发现平台
- 路径推理平台
- 主题聚类平台

## 7. 未来能力预留边界

后续可考虑但本次未实现：

- 路径分析
- 合作圈层分析
- 主题聚类
- 社区发现
- 历史图谱快照与版本回放

这些能力需要额外明确：

- 计算口径
- 数据冻结策略
- 解释方式
- 性能边界

本次仅保留能力边界说明，不把这些未来方向包装为当前已完整上线能力。
