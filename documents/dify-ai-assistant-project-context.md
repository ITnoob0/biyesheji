# Dify AI 助手项目上下文知识库

更新时间：2026-05-03
适用范围：高校教师科研画像与智能辅助系统中的 Dify AI 助手

## 1. 项目定位

本系统是围绕高校教师科研画像主链路建设的教研管理与智能辅助系统。AI 助手的定位是：

- 基于系统已有教师资料、成果、画像、项目推荐、图谱和学院看板上下文提供解释型辅助。
- 基于 Dify 的大模型能力和知识库 RAG 能力增强回答质量。
- 回答必须可解释、可追溯、可回退，不应夸大为通用科研知识平台。

当前更准确的表述是：

- `Dify 增强问答`
- `基于项目知识库和当前页面上下文的智能辅助`
- `围绕教师科研画像主链路的受控 RAG 问答`

不应表述为：

- 完整通用知识平台
- 自动科研评价裁决系统
- 替代科研处或学术委员会的最终认定工具
- 复杂多轮 Agent 工作流平台

## 2. 当前接入链路

### 2.1 前端入口

前端使用悬浮式 AI 助手组件：

```text
frontend/src/components/assistant/FloatingDifyAssistant.vue
```

组件特点：

1. 页面右下角悬浮入口。
2. 支持拖拽位置。
3. 默认提示用户可以询问：本周科研重点、申报准备、成果补强建议。
4. 发送请求到后端接口：

```text
POST /api/ai-assistant/dify-chat/
```

5. 请求体包含：

```json
{
  "message": "用户问题",
  "conversation_id": "Dify 会话 ID，可为空",
  "context_hint": "页面或模块上下文",
  "route_path": "当前前端路由"
}
```

### 2.2 后端入口

后端 Dify 代理入口：

```text
backend/ai_assistant/views.py
DifyAssistantChatView
```

接口要求用户已登录，使用 JWT 鉴权，随后调用：

```text
backend/ai_assistant/dify_service.py
DifyAssistantService.chat(...)
```

### 2.3 后端配置项

Dify 相关环境变量位于 Django settings 中：

```text
DIFY_BASE_URL
DIFY_API_KEY
DIFY_REQUEST_TIMEOUT
```

建议配置示例：

```bash
set DIFY_BASE_URL=https://api.dify.ai/v1
set DIFY_API_KEY=app-xxxxxxxxxxxxxxxx
set DIFY_REQUEST_TIMEOUT=30
```

如果使用私有部署 Dify，则 `DIFY_BASE_URL` 应改为私有部署 API 地址，末尾不要带 `/chat-messages`，后端会自动拼接接口路径。

## 3. Dify 调用方式

后端优先调用 Chat App 接口：

```text
{DIFY_BASE_URL}/chat-messages
```

如果 Chat App 接口不可用，后端再尝试 Workflow App 接口：

```text
{DIFY_BASE_URL}/workflows/run
```

## 4. Chat App 输入变量

当调用 Chat App 时，后端会发送以下输入：

```json
{
  "inputs": {
    "context_hint": "页面上下文",
    "route_path": "当前路由",
    "role": "admin 或 teacher",
    "department": "当前用户学院/部门",
    "title": "当前用户职称"
  },
  "query": "用户问题",
  "response_mode": "blocking",
  "conversation_id": "历史会话 ID",
  "user": "user-当前用户ID"
}
```

Dify 应重点使用以下变量：

- `role`：区分管理员和教师。
- `department`：回答学院看板、教师画像和推荐时用于限定上下文。
- `title`：回答教师发展建议时只作为辅助参考，不得作出绝对评价。
- `context_hint`：当前页面或打开助手时传入的模块提示。
- `route_path`：当前前端路由，可判断用户正在成果、画像、推荐、看板还是通用页面提问。

## 5. Workflow App 输入变量

当调用 Workflow App 时，后端会发送：

```json
{
  "inputs": {
    "draft": "用户问题",
    "current_role": "admin 或 teacher",
    "current_page": "归一化页面",
    "page_context": "页面上下文",
    "answer_style": "简洁执行型",
    "language": "简体中文"
  },
  "response_mode": "blocking",
  "user": "user-当前用户ID"
}
```

`current_page` 归一化规则：

| 前端路由或上下文 | current_page |
| --- | --- |
| 包含 `project-recommendations/preparation` | `project-preparation` |
| 包含 `project-recommendations` | `project-recommendation` |
| 包含 `academy-dashboard` | `academy-dashboard` |
| 包含 `dashboard` | `portrait-dashboard` |
| 包含 `profile-editor` 或 `/entry` | `achievement-center` |
| 其他 | `general` |

## 6. 推荐的 Dify 系统提示词原则

建议在 Dify App 的系统提示词中加入：

```text
你是“高校教师科研画像与智能辅助系统”的 AI 助手。你的回答必须优先基于知识库检索结果、当前页面上下文和用户角色。你不能编造系统未覆盖的规则、政策、成果来源或教师评价结论。

回答优先级：
1. 涉及科研评价、成果录入、积分认定时，优先使用 `research-evaluation-core-rules-final.md`。
2. 涉及政策出处、文件依据时，使用 `research-evaluation-source.txt`。
3. 涉及演示成果为什么这样认定时，使用 `rule-achievement-real-demo-sources.md`。
4. 涉及 Dify 接入、页面上下文、角色差异、回退策略时，使用 `dify-ai-assistant-project-context.md`。
5. 未检索到依据时，明确说明“当前知识库未覆盖”，不要补造。

回答风格：简体中文、简洁执行型、先给结论，再给依据，再给下一步建议。涉及评价规则时必须提示“最终认定仍以学校科研管理部门或学术委员会审核为准”。
```

## 7. 典型场景回答策略

### 7.1 成果是否录入

当用户问“某成果能不能录入/计分”时：

1. 先判断是否属于最终保留的六类成果：科研项目、学术论文与著作、科研成果获奖、成果转化、智库成果、定性评价。
2. 再判断是否属于剔除项：创业计划认证、创新创业大赛、平台与团队、科普基地、科普项目、科普类获奖、成果普及奖、偏教学支撑内容、重大网络传播优秀网络文化成果等。
3. 如果命中保留类，说明可能的规则位置和需要补充的字段。
4. 如果命中剔除类，说明“不录入、不参与总分、不进入能力雷达、不进入年度评价”。
5. 如果无法判断，要求补充成果名称、类别、时间、单位、来源证明和对应文件。

### 7.2 积分为什么这样算

当用户问“为什么这样评价/认定”时：

1. 优先检索最终规则。
2. 如涉及政策原文，再检索来源文件。
3. 如果是演示账号或演示成果，检索真实演示来源清单。
4. 回答中明确：规则条目、成果事实、映射理由、是否保守映射、时间归一说明。

### 7.3 教师画像解释

当用户问画像或雷达维度时：

1. 说明当前画像基于系统内成果数据、聚合统计和规则解释。
2. 不把雷达分数解释为绝对学术水平，只解释为当前系统数据下的结构性画像。
3. 指出可能短板和补强方向时，要用“建议/可优先补充”而非“必须/一定”。

### 7.4 项目推荐解释

当用户问推荐原因时：

1. 结合教师研究方向、成果关键词、成果类型和项目指南匹配说明。
2. 说明当前推荐是规则增强型、可解释推荐，不是复杂机器学习模型结论。
3. 如果推荐证据不足，应提示补充成果关键词、代表成果、项目经历或研究方向信息。

### 7.5 学院看板问答

管理员在学院看板提问时：

1. 允许回答学院范围的聚合分析、趋势、排行和管理建议。
2. 不应暴露超出当前管理员权限的数据。
3. 应提示当前看板是轻量管理分析入口，不是完整 BI 平台。

教师在学院相关场景提问时：

1. 只围绕本人可见信息和公开规则解释。
2. 不输出其他教师个人细节或不属于本人权限范围的数据。

## 8. 回退策略

如果 Dify 未配置：

```text
AI 助手暂未完成配置，请联系管理员补充 Dify 地址与密钥后再试。
```

如果 Dify 网络连接失败：

```text
AI 助手连接失败，请检查 Dify 服务状态后重试。
```

如果 Dify HTTP 服务异常：

```text
AI 助手暂时不可用，请稍后重试。
```

如果 Dify 返回空内容：

```text
助手未返回有效内容，请换个问题再试。
```

知识库回答也应遵守这一思想：不能检索到依据时应明确边界，不要为了回答而编造。

## 9. 建议知识库测试问题

### 9.1 规则类

- `系统最终保留哪些成果录入类型？`
- `哪些内容直接剔除，不录入？`
- `人才计划类项目是否一定剔除？`
- `定性评价在系统里怎么处理？`

### 9.2 来源类

- `这些评价规则来自哪个政策文件？`
- `同一成果符合多项条件时怎么计分？`
- `知识产权未标注延安大学时能不能直接计分？`

### 9.3 演示类

- `曹新亮的项目为什么按国家自然科学基金地区科学基金认定？`
- `李惠的论文为什么可以作为演示论文成果？`
- `周学军专利转让为什么按 14.4 分演示？`
- `杨彩霞智库成果为什么按保守口径映射？`

### 9.4 系统接入类

- `当前项目如何接入 Dify？`
- `教师和管理员调用 Dify 时有什么上下文差异？`
- `我在项目推荐页打开助手，Dify 会收到什么页面信息？`
- `Dify 不可用时后端如何回退？`

## 10. 知识库维护建议

1. 新增规则或修改规则时，更新最终规则文件并重新索引。
2. 新增演示账号或演示成果时，同步补充真实来源清单。
3. 修改 Dify 后端入参、前端助手入口或路由时，同步更新本文档。
4. 每次更新后，至少执行第 9 节的测试问题。
