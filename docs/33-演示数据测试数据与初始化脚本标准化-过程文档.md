# 33-演示数据测试数据与初始化脚本标准化-过程文档

## 1. 本轮目标

建立一套最小可用、可重复执行的演示数据恢复机制，让系统在以下场景中都能快速回到可展示状态：

- 答辩演示
- 联调复现
- 主链路人工回归
- 新环境继续开发

本轮重点不是引入复杂数据平台，而是增强现有初始化命令和脚本入口。

## 2. 扫描结论

本轮扫描了以下内容：

- `backend/users/management/commands/init_demo_teachers.py`
- `backend/scripts/startup_preflight.py`
- `backend/scripts/verify_development_16_24_regression.py`
- `README.md`
- `docs/最终答辩演示步骤-第三轮.md`
- `docs/最终验收清单-第三轮.md`
- `docs/数据口径与风险记录-统一版.md`

扫描后确认当前主要问题：

1. 已有演示初始化命令，但只覆盖教师与成果，未统一覆盖项目指南。
2. 演示账号没有固定的标准输出说明，答辩前仍要靠人工记忆。
3. 缺少“恢复到标准可演示状态”的直接脚本入口。
4. 演示数据、测试数据、真实业务数据的边界文档还不够明确。

## 3. 本轮改动

### 3.1 增强现有演示初始化命令

继续复用 `init_demo_teachers`，没有新起并行命令体系，新增能力：

- `--reset-demo-data`
  - 删除并重建内置演示教师数据
  - 删除并重建内置演示项目指南
- `--print-accounts`
  - 打印管理员与教师演示账号清单
  - 明确默认密码和临时密码口径

### 3.2 标准化演示指南数据

在演示初始化命令中补充了内置项目指南数据，用于稳定支撑：

- 推荐结果展示
- 推荐解释展示
- 问答中的推荐说明
- 管理员项目指南管理页演示

### 3.3 增加标准恢复脚本

新增：

- `backend/scripts/restore_demo_state.py`

脚本顺序：

1. `system_preflight`
2. `migrate`
3. `init_demo_teachers --reset-demo-data --print-accounts`

目标是把环境快速恢复到“标准可演示状态”。

### 3.4 回归脚本接入标准化演示数据

`verify_development_16_24_regression.py` 改为在回归前执行：

- `init_demo_teachers --reset-demo-data`

从而减少“数据库里残留旧演示数据导致回归口径漂移”的风险。

## 5. 数据口径说明

### 演示数据

- 由恢复脚本和演示初始化命令生成
- 用于答辩、演示、联调和主链路人工回归

### 回归测试数据

- 由 Django 测试用例在测试数据库中临时构造
- 与演示数据逻辑上分离

### 真实业务数据

- 不应被当前演示恢复脚本覆盖
- 不应与内置演示账号和内置演示指南混用

## 7. 最小人工验证步骤

1. 进入 `backend`
2. 执行 `python scripts\restore_demo_state.py`
3. 记录命令输出的管理员与教师演示账号
4. 启动后端与前端
5. 用 `100001 / teacher123456` 登录，检查画像、成果、推荐、问答、图谱是否都有演示数据
6. 用 `admin / Admin123456` 登录，检查教师管理、项目指南管理和学院看板是否可演示

## 8. 当前边界

- 当前恢复脚本只覆盖内置演示账号和内置演示指南
- 当前没有引入 fixture 平台、数据仓库或复杂主数据系统
- 当前演示恢复流程仍定位于开发 / 答辩 / 演示环境，不是正式生产数据初始化方案
