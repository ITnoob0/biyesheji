# Dify 知识库导入清单与优先级

更新时间：2026-05-03
适用对象：高校教师科研画像与智能辅助系统的 Dify AI 助手知识库
建议知识库名称：`BiShe-科研评价与智能问答核心知识库`

## 1. 建设目标

本知识库用于支撑项目中的 Dify AI 助手，使助手回答时优先基于本项目真实规则、真实政策来源、真实演示成果和系统当前接入方式进行 RAG 检索，而不是只依赖大模型通用知识。

回答目标：

1. 能解释“为什么这样评价/认定”。
2. 能说明“某类成果是否应录入、是否参与核心科研能力评分”。
3. 能解释“系统当前 Dify 接入边界、页面上下文、教师/管理员角色差异”。
4. 能围绕教师画像、成果结构、项目推荐、学院看板给出受控、可追溯、可回退的回答。

## 2. 必须导入的 P0 文档

以下三个文件必须导入同一个 Dify 知识库，且应保持启用状态。

| 优先级 | 文件路径 | 文档用途 | 检索关键词建议 |
| --- | --- | --- | --- |
| P0 | `documents/research-evaluation-core-rules-final.md` | 核心科研能力评价规则，最重要，是回答评价规则、录入边界、积分认定的第一依据 | 核心科研能力、评价规则、积分、录入范围、剔除项、就高原则、科研项目、论文、成果获奖、成果转化、智库成果、定性评价 |
| P0 | `documents/research-evaluation-source.txt` | 规则来源和政策依据，用于回答规则出处、政策原文、为什么设置这些规则 | 延大发〔2022〕52号、高质量贡献评价与激励标准、科研部分2022版、政策依据、积分标准、总则、认定规则 |
| P0 | `documents/rule-achievement-real-demo-sources.md` | 真实演示成果来源，用于解释演示数据为什么这样评价/认定，支持答辩追问 | 演示成果、真实来源、公开来源、规则条目、PROJECT_NS_06、PAPER_HS_07、AWARD_HS_05、TRANS_09、THINK_05 |

## 3. 建议同时导入的 P1 项目上下文文档

为了让 Dify 助手理解本项目的真实接入方式，还应导入以下补充文档：

| 优先级 | 文件路径 | 文档用途 |
| --- | --- | --- |
| P1 | `documents/dify-ai-assistant-project-context.md` | 说明项目后端、前端、接口、角色、页面上下文和回退策略 |
| P1 | `documents/dify-ai-assistant-rag-playbook.md` | 说明 Dify 知识库切分、检索、提示词和测试问题配置 |

## 4. 建议导入方式

### 4.1 推荐方式：独立文档导入

在 Dify 中新建知识库后，逐个上传上述 P0/P1 文档。这样可以在检索结果中保留来源文件名，便于定位回答依据。

### 4.2 可选方式：打包导入

也可以在项目根目录执行：

```bash
python backend/scripts/build_dify_knowledge_bundle.py
```

生成：

```text
documents/dify-knowledge-base-bundle.md
```

然后将该打包文件导入 Dify。打包文件适合快速导入，但单文件来源粒度不如独立文档清晰。

## 5. Dify 知识库建议配置

### 5.1 知识库类型

建议创建文本知识库，导入 Markdown/TXT 文档。

### 5.2 索引方式

建议选择高质量索引。原因：本项目知识库中既有长规则文本，也有规则编号、成果类别、文件号、人员姓名、项目名称等精确匹配内容，高质量索引更适合后续启用混合检索和重排。

### 5.3 切分方式

建议优先选择父子分段模式：

- 父段：按二级/三级标题形成较完整语义块。
- 子段：按段落、表格行、规则条目切分。
- 父段用于保留完整规则背景，子段用于提升问题命中精度。

如果当前 Dify 版本暂不便设置父子分段，则使用通用分段，分隔符建议为连续换行。

### 5.4 检索策略

建议启用混合检索或语义检索 + 关键词增强，原因是用户问题会同时出现：

- 语义问法：`为什么这个成果可以计分？`
- 规则问法：`国家自然科学基金地区科学基金项目是多少分？`
- 编码问法：`PROJECT_NS_06 为什么认定？`
- 页面问法：`我在项目推荐页，为什么推荐这个指南？`

## 6. 推荐元数据标注

导入时可为文档或分段维护以下元数据字段：

| 字段 | 示例 | 说明 |
| --- | --- | --- |
| `priority` | `P0` / `P1` | 检索优先级 |
| `domain` | `research_evaluation` / `dify_integration` / `demo_sources` | 知识域 |
| `source_type` | `rule_final` / `policy_source` / `demo_source` / `project_context` | 来源类型 |
| `audience` | `teacher` / `admin` / `all` | 适用角色 |
| `answer_scope` | `evaluation_rule` / `project_usage` / `demo_explanation` | 回答范围 |

## 7. 回答优先级规则

Dify 助手生成回答时建议遵守以下顺序：

1. 先引用 `research-evaluation-core-rules-final.md` 判断当前系统最终口径。
2. 如果用户追问来源、政策依据，再引用 `research-evaluation-source.txt`。
3. 如果用户追问演示成果为什么这样设置，再引用 `rule-achievement-real-demo-sources.md`。
4. 如果用户问系统接入、页面、角色、回退方式，再引用 `dify-ai-assistant-project-context.md`。
5. 检索不到依据时，不要编造；应回答“当前知识库未覆盖该问题，需要管理员补充规则或来源”。

## 8. 知识库边界提示

本知识库只服务本项目的智能问答助手，不应包装成通用科研评价平台。

明确不覆盖：

- 其他高校的科研评价规则。
- 未录入或未审核的真实业务数据。
- 未在知识库中出现的政策更新。
- 对教师个人作出超出系统数据依据的主观评价。
- 自动替代学院管理员、科研处或学术委员会的最终认定。

## 9. 最小验收问题

导入知识库后，建议在 Dify 检索测试中至少测试以下问题：

1. `核心科研能力评价系统最终保留哪些成果类型？`
2. `创业计划认证和创新创业大赛为什么不录入？`
3. `国家自然科学基金地区科学基金项目在系统里如何计分？`
4. `李惠的论文为什么映射到 PAPER_HS_07？`
5. `周学军的专利转让为什么按 8/万元计算？`
6. `如果 Dify 没有返回答案，项目后端会怎么回退？`
7. `教师和管理员调用 Dify 时输入上下文有什么区别？`
8. `我在项目推荐页提问时，助手应该如何结合页面上下文回答？`

## 10. 维护规则

1. 规则最终口径变更时，优先更新 `research-evaluation-core-rules-final.md`。
2. 政策依据变更时，同步更新 `research-evaluation-source.txt` 或新增来源文件。
3. 演示数据变更时，同步更新 `rule-achievement-real-demo-sources.md`。
4. Dify 接口、输入变量、前端助手入口变化时，同步更新 `dify-ai-assistant-project-context.md`。
5. 每次更新后，重新上传或同步 Dify 知识库，并执行最小验收问题。
