# 37-演示账号密码保留与初始化脚本收口-过程文档

## 1. 本轮目标

解决两个实际问题：

1. 初始化和回归脚本在验证时频繁改动演示账号密码
2. 过程文档 `35`、`36` 分叉，影响后续查看现状

本轮重点是把账号密码策略改为“默认保留、显式重置”，并把相关文档收口。

## 2. 问题原因

### 2.1 为什么之前会改掉账号密码

根因在两个地方：

- [backend/users/management/commands/init_demo_teachers.py](e:/Project/BiShe/backend/users/management/commands/init_demo_teachers.py)
  - 之前每次执行都会对管理员和演示教师调用 `set_user_password(...)`
- [backend/scripts/verify_development_16_24_regression.py](e:/Project/BiShe/backend/scripts/verify_development_16_24_regression.py)
  - 之前回归脚本会先执行 `init_demo_teachers --reset-demo-data`
  - 然后再用固定明文密码去调用 `/api/token/`

这套设计对“标准化演示环境”是方便的，但对用户已经设置好的密码不友好，会让数据库里的现有密码被覆盖。

### 2.2 关于“去数据库里查看密码”

数据库里保存的是密码哈希，不是明文密码。  
因此系统可以做到：

- 检查账号是否有可用密码哈希
- 检查某个候选密码是否匹配

但不能直接从数据库里把原明文密码“读出来”显示。

## 3. 本轮修改

### 3.1 初始化命令默认保留密码

更新：

- [backend/users/management/commands/init_demo_teachers.py](e:/Project/BiShe/backend/users/management/commands/init_demo_teachers.py)

当前规则：

- 默认执行 `python manage.py init_demo_teachers`
  - 保留数据库中已有账号密码
  - 只同步资料、成果、演示指南等数据
- 仅在以下情况下才设置密码：
  - 新建账号
  - 账号没有可用密码
  - 显式传入 `--reset-passwords`

### 3.2 新增显式密码重置开关

新增参数：

- `--reset-passwords`

用于明确表达“这次就是要把演示账号密码重置成标准演示密码”，避免脚本隐式覆盖用户现有密码。

### 3.3 恢复脚本默认改为安全模式

更新：

- [backend/scripts/restore_demo_state.py](e:/Project/BiShe/backend/scripts/restore_demo_state.py)

当前默认行为：

- 运行迁移
- 同步演示数据
- 默认保留现有密码

如确实需要恢复成“统一默认密码 + 标准演示数据”，需显式使用：

- `--hard-reset-demo-data`
- `--reset-passwords`

### 3.4 主链路回归脚本不再依赖硬编码密码

更新：

- [backend/scripts/verify_development_16_24_regression.py](e:/Project/BiShe/backend/scripts/verify_development_16_24_regression.py)

当前改为：

- 不再通过固定明文密码调用 `/api/token/`
- 改为使用 `force_authenticate(...)` 执行保护接口主链路验证
- 对账号状态改为校验：
  - 账号存在
  - 账号激活
  - 数据库中存在可用密码哈希

这样做的好处是：

- 不会再为了回归验证覆盖你数据库里的现有密码
- 仍然可以保证主链路回归脚本稳定可重复执行

### 3.5 演示初始化测试同步补齐

更新：

- [backend/users/tests_demo_init.py](e:/Project/BiShe/backend/users/tests_demo_init.py)

新增验证：

- 默认执行初始化命令时，已有账号密码应被保留
- 显式传入 `--reset-passwords` 时，才恢复标准演示密码
