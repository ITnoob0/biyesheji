# 91-成果认领通知与学院看板展示优化记录

## 1. 本次优化背景

本次检查集中处理两个展示与口径问题：

- 成果认领有时仍会出现在右上角通知中，容易被误解为当前规则化成果申报入口。
- 学校/学院管理员看板中成果数量、教师人数与积分指标并列展示，视觉上容易把辅助数量看成评价得分；学院管理员侧教师分析职责需要从总览中拆分出来，职称筛选也需要同步新职称目录。

## 2. 成果认领逻辑说明

当前系统中“成果认领”不是规则化成果申报、审批、计分的主流程。它来源于历史论文合著协作链路：

1. 教师录入旧论文成果时，如果合作作者绑定了校内教师，后端会生成 `AchievementClaim` 认领记录。
2. 系统向目标教师写入 `ACHIEVEMENT_CLAIM` 或 `CLAIM_REMINDER` 通知。
3. 右上角通知中心可能展示“成果认领邀请”或“成果认领提醒”。
4. 当前规则化成果主流程已经转为“教师本人申报 -> 学院管理员审核 -> 审核通过后 final_score 生效”，认领不再作为新规则成果申报入口。

本次前端对通知中心做了清晰化处理：成果认领类通知显示“历史协作”标识，点击时给出历史论文合著协作说明，并引导用户进入当前成果查看或审核入口，避免跳转到旧的 `/profile-editor/achievement-claims` 路径。

涉及文件：

- `frontend/src/layout/MainLayout.vue`
- `backend/achievements/services.py`
- `backend/users/models.py`

## 3. 学院看板展示优化

调整后的看板口径如下：

- 积分卡片只展示评价积分：核心科研积分、学术产出积分、项目竞争积分、奖励转化积分、平台科普积分。
- 成果数量作为对应积分卡片的辅助说明展示为“X 项”，不再单独占用一排卡片。
- 教师总数、唯一成果数量、待审核成果数量放在顶部范围摘要中，便于管理员先判断统计范围。
- 总览页只保留顶部数据总览和年度积分趋势图，不再显示教师速览、近期成果和审批提醒等下方明细卡片。
- 年度积分趋势图使用积分与数量双轴展示，并在悬浮提示中补充学术产出、科研项目、奖励转化、平台科普等年度结构信息。

涉及文件：

- `frontend/src/views/AcademyDashboardView.vue`
- `backend/achievements/management_dashboard.py`

## 4. 学院管理员菜单与职称筛选

教师分析子菜单保留并承担教师明细入口职责：系统管理员可查看全校教师，学院管理员可查看本院教师。该页展示当前权限范围内所有教师，并以页面弹窗方式提供教师画像、全部成果、趋势分析可视化入口。

趋势分析弹窗使用当前规则化科研成果口径，不再沿用旧“论文、项目、知识产权、学术服务”展示标签。近年成果结构统一展示为学术产出、科研项目、奖励转化、平台科普。

画像维度卡片说明同步收口为“纳入成果大类 + 原始积分到画像展示分转换公式”两层口径。具体为：学术产出与著作纳入论文、著作、译著、工具书、专辑、歌曲等学术产出；项目竞争与资源获取纳入科研项目；成果获奖与学术荣誉纳入科研成果获奖；转化服务与智库贡献纳入成果转化、智库成果；平台团队与科普影响纳入平台与团队、科普类获奖。画像展示分只由对应维度原始积分换算，不使用代表成果数、成果项数或金额基数作为加成，避免把数量误读为评价导向。例如学术产出与著作为 `min(论文著作原始积分 ÷ 30, 100)`。

职称筛选统一使用后端标准目录，当前标准职称包括：

- 教授
- 副教授
- 讲师
- 助教
- 研究员
- 副研究员
- 助理研究员
- 研究实习员

## 5. 测试关注点

本次补充后端用例覆盖以下内容：

- 学院看板接口返回的职称筛选项包含后续新增的标准职称。
- 核心科研积分卡片辅助说明包含教师总数、唯一成果数量和待审核数量。
- 各大类积分卡片只展示“X 项”形式的成果数量文案。
- 教师分析页通过全量排行参数返回当前权限范围内所有教师。
- 原有成果认领接口仍不作为当前暴露路由，避免重新打开旧申报链路。
