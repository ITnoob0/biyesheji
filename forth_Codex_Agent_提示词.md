# 完全版后续开发任务拆解与 Codex Agent 提示词

## 一、使用说明

这份文档基于《项目完全版后续开发目标》整理，用于把“完整版本”目标拆成一组可以逐项推进的开发任务，并为每个任务生成一份可以直接复制给 Codex 的 Agent 提示词。

适用方式：

* 不要一次性把全部任务都交给 Codex。
* 建议一次只推进一个主目标模块。
* 每个任务执行时，都要求 Codex：

  * 先扫描仓库；
  * 再做最小必要增量修改；
  * 同步补文档与验证步骤；
  * 明确区分历史问题与本轮新增问题。

本文档对应的整体方向是：

* 继续基于当前系统增量开发；
* 不推翻重做；
* 把“系统最终形态”作为方向；
* 把“最小可落地”作为执行原则。

## 二、通用总约束模板（建议每个任务都附在前面）

下面这段建议作为所有任务提示词的公共前缀。

```text
你现在是本项目的后续开发 Agent，请直接在当前仓库中完成本次任务相关修改。你的任务不是泛泛给建议，而是基于现有系统做最小必要、可验证、可维护的增量开发。

【总原则】
- 必须基于现有系统增量开发，不允许推翻重做。
- 登录态继续使用 sessionStorage，不要改成别的会话方案。
- 教师账号继续使用工号登录，不要改成邮箱登录或别的主认证方式。
- Neo4j 不能成为论文录入和教师画像展示的强依赖，必须保留 MySQL 回退链路。
- 所有新增页面和新增路由必须接入现有严格路由守卫。
- 新增功能时，优先考虑职责拆分；如果一个文件同时包含页面渲染、表单状态、数据库读写、业务规则和工具函数，应拆成多个模块。
- 不要把“预留能力”描述成“已完整上线能力”。
- 不要为了“完整版本”做大规模架构重构、拆库、换框架或无收益的新表扩张。
- 优先复用现有模块、服务层、请求层、模型层、类型层和页面结构。

【工作方式】
1. 先扫描仓库，再动手修改。
2. 从 package.json、lock 文件、requirements、pyproject.toml、manage.py、脚本文件中识别真实技术栈与命令，不要凭空假设。
3. 小步修改，逐项验证。
4. 同步更新文档、验证说明和必要测试。
5. 明确区分历史遗留问题与本次新增问题。
6. 生成本次开发的过程性文档，保存在docs文件夹内。

【验证要求】
完成后尽量运行并记录实际执行的验证命令，包括但不限于：
- 前端 type-check
- 前端测试或验证脚本
- 后端 manage.py check
- 后端测试
- 本任务对应主链路的最小人工验证步骤

【最终输出格式】
最终请按以下结构汇报：
1. 变更总览
2. 本任务完成内容
3. 关键文件清单
4. 重要设计决策与权衡
5. 验证命令与结果
6. 当前未完成但已明确边界的部分
7. 已知风险与后续建议

现在开始：先扫描仓库并给出简明实施计划，然后按要求直接修改代码、补文档、补测试并完成验证。
```

## ~~任务 1：认证与账户体系增强~~

### 任务拆解

目标：把当前已具备的工号登录、注册、忘记密码、统一失效跳转，进一步提升为更完整的账户生命周期管理与权限边界体系。

建议本任务聚焦：

* 管理员重置教师账户能力
* 密码修改、安全提示、异常登录提示
* 更清晰的角色身份展示
* 管理员 / 教师权限边界补强
* 多角色扩展的权限模型说明
* 权限边界测试补充

本任务不要做：

* 不改工号登录主流程
* 不切换 sessionStorage
* 不重做整套认证架构
* 不为了未来多角色扩展，提前做过重权限系统重构

### 可直接复制给 Codex 的提示词

```text
你现在要完成的任务是：认证与账户体系增强。

【任务目标】
在当前已经支持工号登录、注册、忘记密码、统一失效跳转的基础上，继续补强账户生命周期管理、角色身份展示、权限边界和账户安全提示机制。

【必须完成】
- 梳理当前账户相关链路，包括登录、注册、忘记密码、密码修改、管理员管理教师账户等。
- 补充或完善管理员重置教师账户能力。
- 补充或完善密码修改能力及相关安全提示。
- 增强异常登录、失效恢复、权限不足时的提示一致性。
- 在前后端界面或接口中更清晰地体现教师与管理员的角色身份。
- 补齐更多角色场景下的权限测试。
- 在文档中补充“当前权限模型说明”和“后续多角色扩展预留说明”。

【实施原则】
- 不改变工号登录主流程。
- 不改变 sessionStorage 登录态方案。
- 优先在现有认证模块、用户模块、权限判断模块、守卫层和管理页面上增量修改。
- 尽量通过统一权限入口和统一错误提示收口，而不是到处散落条件判断。

【验收要求】
- 教师与管理员的权限边界更清楚。
- 管理员可稳定执行教师账户相关管理操作。
- 密码修改、账户异常、权限不足时有清晰提示。
- 关键权限场景具备最小测试或验证脚本。
- 文档明确当前权限边界和未来扩展边界。

【特别禁止】
- 不要重写整套认证系统。
- 不要改成邮箱登录。
- 不要为了“更通用”引入复杂 RBAC 大重构，除非仓库现状证明这是最小必要方案。

请直接在仓库中完成代码、测试、文档与验证说明修改。
```

## 五、这套提示词的核心意图

这套提示词的重点不是让 Codex 进行一次失控的大重构，而是：

* 让它围绕“完整版本目标”逐项推进；
* 每次只处理一个模块；
* 始终坚持增量开发；
* 同步更新文档与验证说明；
* 把“系统最终形态”当方向，把“最小可落地”当执行原则。
