# 一期成果管理闭环增强 过程文档

## 1. 任务背景
- 项目阶段：`V1.1-portrait-baseline`
- 任务目标：在不破坏现有论文主工作流的前提下，为 `Project`、`IntellectualProperty`、`TeachingAchievement`、`AcademicService` 补齐最小可用前后端闭环。
- 硬约束：
  - 不重写认证系统
  - 不把 `sessionStorage` 改回 `localStorage`
  - 教师登录账号继续使用工号
  - Neo4j 不能成为成果录入和画像展示的强依赖

## 2. 开发前扫描结论
### 已确认的现状
- 后端模型已存在五类成果：
  - `Paper`
  - `Project`
  - `IntellectualProperty`
  - `TeachingAchievement`
  - `AcademicService`
- 前端已存在成果录入页入口：
  - [AchievementEntryView.vue](/e:/Project/bishe/frontend/src/views/AchievementEntryView.vue)
- 路由和认证守卫已就位：
  - [index.ts](/e:/Project/bishe/frontend/src/router/index.ts)
  - [sessionAuth.ts](/e:/Project/bishe/frontend/src/utils/sessionAuth.ts)
- 后端成果接口已具备统一模式基础：
  - [views.py](/e:/Project/bishe/backend/achievements/views.py)
  - [serializers.py](/e:/Project/bishe/backend/achievements/serializers.py)

### 发现的问题
- 成果中心页面单文件同时承载：
  - 页面渲染
  - tab 类型
  - 接口地址
  - 选项常量
  - 表单与列表逻辑
- 后端测试对四类新成果的覆盖不够完整，主要集中在论文与部分创建链路。

## 3. 最小可行实现方案
- 保持现有“统一成果中心”页面结构，不额外引入复杂工作台。
- 复用现有 `ModelViewSet + 当前教师隔离` 模式，不新增复杂通用引擎。
- 对前端仅做轻量职责拆分：
  - 类型定义独立
  - 常量与接口映射独立
- 对后端补充接口级测试，覆盖：
  - 创建
  - 列表隔离
  - 删除

## 7. 验证过程
### 自动化验证
- `npm run type-check`
- `python manage.py check`
- `python manage.py test achievements --keepdb`

### 手工验证建议
1. 教师登录系统
2. 进入 `/entry`
3. 分别录入四类新增成果
4. 检查列表是否即时展示
5. 删除一条记录并确认列表更新
6. 确认论文 tab 功能未受影响

## 8. 结果与取舍
### 已达成
- 四类成果已纳入当前一期成果管理闭环
- 权限隔离仍按当前教师生效
- 论文主工作流保持可用

### 主动未做
- 未引入动态表单引擎
- 未引入新的状态管理体系
- 未做大规模页面重构
- 未扩展成果编辑页

## 9. 风险与遗留问题
- 仓库存在部分中文编码噪音，后续文本匹配和补丁修改仍有风险。
- 当前四类成果已有最小闭环，但筛选、排序、编辑能力仍可继续增强。

## 10. 是否纳入当前阶段验收
- 结论：`纳入当前阶段验收`
- 原因：该任务直接增强了“一期成果录入与管理”主链路。
