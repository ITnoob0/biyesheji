# 项目指南管理与推荐 一期过程文档

## 1. 任务背景

本轮任务目标是在当前项目中新增“项目指南管理与推荐”的一期版本，但必须遵守以下约束：

- 属于当前项目的扩展方向，不是既有主链路能力
- 必须采用最小可用实现
- 不引入 RAG、向量数据库或复杂代理流程
- 推荐结果应强调“可解释”，而不是复杂算法

因此本轮只实现两部分：

- A. 项目指南数据管理
- B. 基于教师画像与标签的轻量推荐

## 2. 设计取舍

### 2.1 为什么新增独立模块

当前仓库中尚不存在 `funding_call`、`recommendation` 等模型。  
如果直接把项目指南和推荐逻辑硬塞进 `achievements`，会让成果模块承担过多职责。

因此本轮采用独立轻量模块：

- `backend/project_guides`

这样做的原因：

- 不污染当前已稳定的成果主链路
- 更容易控制扩展边界
- 后续若接入智能推荐或 RAG，可在该模块继续演进

### 2.2 为什么不新增推荐结果表

本轮没有新增推荐结果持久化表。

原因：

- 一期推荐规则足够轻量，可实时计算
- 实时计算更容易解释与调试
- 避免过早引入推荐快照、任务重算和结果过期管理

## 3. 一期数据模型

本轮新增模型：

- `ProjectGuide`

核心字段包括：

- 指南标题
- 发布单位
- 指南级别
- 状态
- 截止时间
- 指南摘要
- 主题关键词
- 面向学科/方向
- 资助强度
- 申报要求
- 来源链接
- 创建人
- 创建/更新时间

该模型已经足够支撑：

- 管理员维护指南
- 教师侧轻量规则推荐
- 后续扩展更复杂推荐能力

## 4. 一期推荐规则

推荐规则全部采用可解释的实时打分：

1. 研究方向关键词匹配
   - 教师研究方向标签
   - 教师研究兴趣
   - 论文关键词
   - 与指南 `target_keywords` 做重合判断

2. 学科/院系方向匹配
   - 教师学科信息
   - 教师院系
   - 与指南 `target_disciplines` 做匹配

3. 成果活跃度匹配
   - 统计教师近三年成果数量
   - 若近三年有持续成果，则提升推荐分

4. 申报窗口与资助信息补充
   - 若指南仍处于申报期，增加解释理由
   - 若指南明确资助强度，作为补充说明

每条推荐都返回：

- 推荐分
- 推荐理由
- 命中的关键词
- 命中的学科/方向

本轮明确不做：

- 外部政策抓取
- 复杂语义检索
- 多轮推理推荐

## 7. 后续扩展接口预留

本轮虽然不接入智能推荐或 RAG，但已做以下预留：

1. 推荐逻辑集中在 `services.py`
   - 后续可替换为更复杂策略
   - 当前前后端接口无需重写

2. 推荐接口输出 `data_meta.future_extension_hint`
   - 明确保留后续接入智能推荐能力的空间

3. 指南模型中保留：
   - `summary`
   - `target_keywords`
   - `target_disciplines`
   - `source_url`
   这些字段后续都可用于语义增强或 RAG 检索
