# 第二轮主链路收口与第三轮边界整理-过程文档

最后更新时间：2026-03-18

## 一、任务背景

本轮开发定位不是推翻重做，也不是提前做完第三轮所有高级能力，而是围绕以下目标开展：

1. 收口主链路，提升稳定性和可维护性。
2. 清理接口、数据、模块和文档边界。
3. 为第三轮展示增强和能力补足留下清晰承接条件。

本轮继续遵守以下硬约束：

- 认证状态继续使用 `sessionStorage`。
- 教师仍使用工号登录。
- Neo4j 不能成为论文录入和画像展示的强依赖。
- 新增或改造功能继续接入现有路由守卫。
- 不把推荐、看板二期、RAG、复杂图分析包装成本轮已完成核心能力。

## 二、本轮完成内容

### 1. 认证与会话收口

已完成：

- 新增统一请求层：[frontend/src/utils/http.ts](e:/Project/bishe/frontend/src/utils/http.ts)

  - 登录后优先回跳到失效前页面

效果：

- 登录态过期时，各页面行为更一致。
- 受保护页面不会静默失败或无提示卡死。

## 四、建议的人工验证步骤

### 1. 认证与会话

1. 使用教师工号登录。
2. 进入画像页、成果页、图谱页。
3. 手动让 token 失效，或在浏览器里清除 access token 后刷新页面。
4. 观察是否统一跳回登录页并提示“登录状态已失效”。
5. 重新登录后，确认是否回到原来尝试访问的页面。

### 2. 成果管理

1. 进入成果页。
2. 新增一条论文记录。
3. 在列表中通过题目关键字搜索到该记录。
4. 点击“编辑”，修改题目、期刊和论文类型。
5. 保存后确认列表即时更新。
6. 在项目、知识产权、教学成果、学术服务页重复执行一次“搜索 + 编辑”。
7. 确认 BibTeX 导入链路仍可正常打开与导入。

### 3. 图谱链路

1. 录入一条论文或项目数据。
2. 打开教师画像页中的学术拓扑图。
3. 观察右侧“数据来源”区块，确认是否显示 Neo4j / MySQL 状态。
4. 在 Neo4j 不可用时，确认图谱页面仍返回 MySQL 回退数据或明确空状态。
5. 确认论文录入与画像展示不会因为 Neo4j 异常而中断。

### 4. 用户信息同步

1. 教师进入资料页修改姓名、学院、职称、学科、研究兴趣。
2. 返回画像页确认显示同步更新。
3. 管理员进入教师管理页修改同一教师资料。
4. 再进入画像页与资料页，确认字段一致。

## 六、取舍说明

### 本轮主动做的取舍

- 没有把成果页重写成全新的列表页 + 详情页体系，而是复用现有表单布局做编辑增强。
- 没有把 `CustomUser` 与 `TeacherProfile` 合表，而是用统一服务入口收口同步逻辑。
- 没有让 Neo4j 成为硬依赖，而是保留并强化 MySQL 回退展示。
- 没有直接做前端测试框架大建设，而是先补共享类型、关键后端测试和可复现验证步骤。

### 这样做的原因

- 符合“第二轮以最小必要增量开发为主”的目标。
- 保持现有主链路稳定，避免为了结构理想化而破坏答辩演示。
