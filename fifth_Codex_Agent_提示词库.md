# P0 级任务整理与 Codex Agent 开发提示词

## 一、P0 级任务定位

P0 不是继续堆新功能，而是优先补齐那些会直接影响**稳定交付、稳定演示、稳定维护**的基础能力。它们决定系统能不能作为一个可持续开发、可答辩、可验收的企业级项目继续推进。包括：

1. 数据库迁移执行与启动前检查机制
2. 最小 CI / 自动化验证流水线
3. 统一日志、异常提示与错误回退治理
4. 统一接口清单、模块边界、数据口径与交付文档持续维护机制
5. 管理员与教师权限边界进一步收口
6. 演示数据、测试数据与初始化脚本标准化
7. 主链路回归脚本持续扩充

## 三、P0 任务 1：启动前检查与迁移治理

### 任务拆解

目标是防止再次出现“代码已经读取新字段，但数据库还没迁移”这类问题。后续方向包括：启动前检查迁移是否已执行、检查关键依赖是否安装、检查 MySQL 连接与必要表结构、明确迁移执行顺序、把迁移检查纳入演示脚本。

### 可直接复制给 Codex 的提示词

```text
你现在要完成的任务是：启动前检查与迁移治理。

【任务目标】
在当前系统多轮增量开发的基础上，建立最小可用的启动前检查与迁移治理机制，重点防止“代码已读新字段，但数据库未迁移”之类的问题再次出现。

【必须完成】
- 扫描当前后端启动方式、manage.py、配置文件、README、部署说明和已有脚本。
- 梳理当前数据库迁移执行方式和启动依赖。
- 增加启动前检查机制，至少覆盖以下方面：
  - 检查迁移是否已执行
  - 检查关键依赖是否安装
  - 检查 MySQL 连接是否可用
  - 检查必要表结构是否存在
- 增加或完善迁移执行顺序说明。
- 在演示脚本或启动文档中加入迁移检查步骤。
- 若仓库已有类似脚本，优先增强而不是重复新建一套并行方案。

【实施原则】
- 优先使用现有 Python / Django 管理命令、启动脚本、文档体系。
- 优先做最小可执行、可理解、可维护的检查机制。
- 不要为了“企业级”重构整个部署体系。

【验收要求】
- 启动前能够检查关键迁移与依赖状态。
- 文档中明确启动前检查和迁移执行顺序。
- 演示或初始化流程中包含迁移检查步骤。
- 若检查失败，输出信息可理解、可操作。

【特别禁止】
- 不要引入复杂部署平台。
- 不要重写整个启动流程。
- 不要做与当前仓库技术栈不一致的方案。

请直接在仓库中完成代码、脚本、文档和验证说明修改。
```

## 四、P0 任务 2：最小 CI 与自动化验证流水线

### 任务拆解

目标是让关键验证不再依赖人工记忆。后续方向包括接入最小 CI，自动执行后端 `manage.py check`、后端关键测试、前端 `verify:baseline`、统一主链路回归脚本，并为失败结果提供最小可读日志。

### 可直接复制给 Codex 的提示词

```text
你现在要完成的任务是：最小 CI 与自动化验证流水线。

【任务目标】
为当前系统建立最小可用的 CI / 自动化验证流水线，让关键验证不再依赖人工记忆，形成企业级持续开发的最小防回归装置。

【必须完成】
- 扫描当前仓库中的前端命令、后端命令、测试脚本、基线验证脚本、CI 配置文件和文档。
- 识别真实存在且可运行的验证命令。
- 建立最小 CI 或自动化验证流水线，至少自动执行：
  - 后端 manage.py check
  - 后端关键测试
  - 前端 verify:baseline 或等价基线脚本
  - 统一主链路回归脚本
- 为失败结果提供最小可读日志或输出说明。
- 在文档中增加“本地验证顺序”和“CI 验证范围”说明。

【实施原则】
- 优先使用仓库现有脚本，不要重新发明一套命令体系。
- 以最小可跑通为原则，不追求一次性覆盖全部测试场景。
- 若当前仓库没有正式 CI 配置，则至少落地一个标准化自动验证入口，并附带后续接入 CI 的说明。

【验收要求】
- 至少有一条统一自动验证链路可用。
- 关键检查项能够被自动执行。
- 失败时输出可读。
- 文档能指导他人重复执行。

【特别禁止】
- 不要凭空假设 npm / pnpm / yarn / pytest / unittest，必须以仓库真实配置为准。
- 不要为了 CI 重构大量业务代码。
- 不要追求大而全流水线，先做最小闭环。

请直接完成 CI / 脚本 / 文档 / 验证说明修改。
```

## 五、P0 任务 3：统一日志、异常提示与错误回退治理

### 任务拆解

目标是避免接口异常直接暴露调试页，避免前后端错误提示口径碎片化。后续方向包括统一后端错误响应结构、统一前端异常展示口径、对登录/推荐/问答/图谱回退做统一降级说明、明确开发环境和正式环境的异常展示差异、补系统级错误排查说明。

### 可直接复制给 Codex 的提示词

```text
你现在要完成的任务是：统一日志、异常提示与错误回退治理。

【任务目标】
让当前系统在出现异常时具备更一致、更可理解、更可恢复的表现，避免接口异常直接暴露调试页，也避免前后端错误提示口径碎片化。

【必须完成】
- 扫描当前后端异常处理、中间件、接口返回结构、前端请求层、错误提示方式、降级展示和回退逻辑。
- 梳理当前最常见的异常类型和错误提示分布。
- 统一后端错误响应结构。
- 统一前端异常展示口径。
- 对以下关键链路增加统一降级说明或回退提示：
  - 登录
  - 推荐
  - 问答
  - 图谱
- 明确开发环境与正式环境的异常展示差异，并在文档中写清。
- 补充系统级错误排查说明。
- 若已有统一请求层或异常层，优先在该层收口。

【实施原则】

- 优先收口高风险主链路。
- 保持现有主链路行为稳定，避免因统一异常处理引入新的静默失败。

【验收要求】
- 前后端错误提示更一致。
- 关键链路异常时用户可理解、可恢复。
- 正式环境不再轻易暴露调试信息。
- 文档中有错误排查和环境差异说明。

【特别禁止】
- 不要为了日志治理引入过重的运维平台。
- 不要大改所有接口契约，除非这是最小必要调整。
- 不要让统一异常处理破坏已有正常响应链路。

请直接完成代码、文档、验证说明和必要测试。
```

```text
【验收要求】

【特别禁止】

- 不要让文档与代码脱节。
- 不要把未来能力写成当前已完成能力。

请直接完成文档整理、必要索引更新和验证说明补充。
```

## 九、P0 任务 7：主链路回归脚本持续扩充

### 任务拆解

这项是 P0 总览中明确提出的基础项，目标是让系统最关键的演示闭环和主业务闭环具备可重复验证能力。它与最小 CI 一起构成企业级持续开发的“防回归底座”。

### 可直接复制给 Codex 的提示词

```text
你现在要完成的任务是：主链路回归脚本持续扩充。

【任务目标】
围绕当前系统最关键的业务闭环，持续扩充主链路回归脚本，让核心流程具备可重复、可执行、可记录的验证基础，减少回归依赖人工记忆。

【必须完成】
- 扫描当前已有回归脚本、验证脚本、README、演示文档和测试说明。
- 梳理当前最关键的主链路，包括但不限于：
  - 登录与失效回跳
  - 教师资料查看与编辑
  - 成果新增 / 编辑 / 查询
  - 画像展示
  - 图谱展示与回退
  - 推荐查看
  - 管理员关键操作
- 在现有基础上扩充统一主链路回归脚本。
- 若无法全部自动化，至少形成准自动化脚本 + 标准验证说明。
- 在文档中明确每条主链路的验证目的、前置条件和预期结果。
- 与最小 CI 或统一验证入口保持兼容。

【实施原则】
- 优先覆盖高风险、高频回归链路。
- 优先使用仓库现有脚本和命令风格。
- 自动化能力不足时，先保证脚本化和标准化。

【验收要求】
- 核心主链路比当前更容易重复验证。
```
