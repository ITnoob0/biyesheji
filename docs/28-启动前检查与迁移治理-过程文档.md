# 启动前检查与迁移治理 - 过程文档

## 1. 任务背景

当前系统已经经历多轮增量开发，数据库字段和迁移数量持续增加。  
此前已实际出现过以下问题：

- 代码已经读取新字段
- 但数据库未执行对应迁移
- 登录接口直接触发后端 500
- 前端把 Django Debug 调试页错误内容暴露到界面上

这说明系统已经需要一个最小可用的“启动前检查与迁移治理”机制。

## 2. 本次扫描结果

本次扫描了：

- `backend/manage.py`
- `backend/core/settings.py`
- `README.md`
- `docs/最终答辩演示步骤-第三轮.md`
- `backend/scripts/`
- `backend/users/management/commands/`

结论如下：

- 当前后端启动方式仍然是标准 `manage.py runserver`
- 当前已有 `init_demo_teachers` 命令和若干脚本
- 当前没有正式的启动前预检查命令
- 当前 README 启动步骤只有：
  - `migrate`
  - `init_demo_teachers`
  - `runserver`
- 当前演示文档中也没有把迁移检查明确纳入演示前准备步骤

## 3. 本次落地方案

### 3.1 设计原则

本次没有重写启动流程，而是采用：

- 复用 Django 管理命令
- 复用现有脚本目录
- 用一个轻量脚本在 Django 初始化前先检查关键依赖
- 用一个 Django 命令负责数据库、迁移和表结构检查

### 3.2 新增内容

新增文件：

- `backend/core/startup_preflight.py`
- `backend/users/management/commands/system_preflight.py`
- `backend/scripts/startup_preflight.py`
- `backend/users/tests_startup_preflight.py`

并增强：

- `backend/scripts/verify_development_16_24_regression.py`
- `README.md`
- `docs/最终答辩演示步骤-第三轮.md`

## 4. 检查覆盖范围

当前启动前检查至少覆盖以下内容：

### 4.1 关键依赖是否安装

检查：

- `Django`
- `djangorestframework`
- `djangorestframework-simplejwt`
- `django-cors-headers`
- `PyMySQL`

说明：

- 依赖检查被放在 `backend/scripts/startup_preflight.py` 中优先执行
- 这样即使 Django 本身还没成功初始化，也能先给出可理解的安装提示

### 4.2 MySQL 连接是否可用

检查：

- `backend/core/settings.py` 中配置的默认数据库
- 实际连接是否可用
- `SELECT 1` 探活

### 4.3 迁移是否已执行

检查：

- 当前数据库是否存在未执行迁移
- 若存在，则列出未执行迁移并提示执行 `python manage.py migrate`

### 4.4 必要表结构是否存在

检查：

- `django_migrations`
- `users_customuser`
- `achievements_teacherprofile`
- `achievements_paper`
- `project_guides_projectguide`

说明：

- 这些表足以覆盖当前登录、教师资料、成果和推荐等主链路

## 5. 当前推荐启动顺序

当前推荐顺序已经统一为：

1. `python scripts\startup_preflight.py`
2. `python manage.py migrate`
3. `python scripts\startup_preflight.py`
4. `python manage.py init_demo_teachers`
5. `python manage.py runserver`

这样做的好处：

- 启动前先看清依赖、数据库和迁移状态
- 迁移完成后再复核一次
- 最后再初始化演示账号和启动服务

## 6. 与现有验证体系的衔接

本次还把新预检查命令并入了：

- `backend/scripts/verify_development_16_24_regression.py`

这样统一回归脚本在跑接口链路前，也会先执行启动前检查。

覆盖内容：

- 当前测试环境下预检查结果应全部通过
- 必要表结构清单应包含核心表

- `python manage.py system_preflight`
- `python scripts\startup_preflight.py`
- `python scripts\verify_development_16_24_regression.py`

## 8. 历史遗留问题与本次新增问题区分

### 历史遗留问题

- 仓库当前仍没有正式 Python 依赖清单文件
- CI 工作流当前仍未建立

### 本次解决的问题

- 启动前缺少依赖、迁移、数据库和表结构检查
- 启动文档未明确迁移检查顺序
- 演示流程未纳入迁移检查步骤

## 9. 后续建议

后续继续增强时，建议优先：

1. 补正式依赖清单
2. 将 `startup_preflight.py` 和关键测试纳入 CI
3. 若以后多环境部署，再为预检查补环境模式说明
