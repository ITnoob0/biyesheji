# 47-学术图谱与图分析增强-过程文档

## 1. 本次任务目标

在保留现有“Neo4j 优先读取 + MySQL 回退链路 + 轻量图分析”主链路的前提下，补齐一批更有业务价值、但仍可解释可回退的图谱增强能力，让图谱不再只是静态可视化，而是可以支撑关系说明、成果回跳和合作圈层观察。

本次交付坚持以下边界：

- Neo4j 仍然只是增强能力，不是主链路强依赖。
- MySQL 回退链路必须保持可用。
- 图分析继续维持轻量增强，不引入社区发现、复杂图挖掘、图机器学习等重型方案。
- 所有分析结果都必须带有解释说明、降级说明和边界提示。

## 2. 扫描结果与当前基线

本次扫描了以下图谱相关实现：

- 前端图谱页面：`frontend/src/views/AcademicGraph.vue`
- 图谱类型定义：`frontend/src/types/graph.ts`
- 图谱来源说明：`frontend/src/views/graph/sourceState.js`
- 后端图谱接口：`backend/graph_engine/views.py`
- 后端图分析服务：`backend/graph_engine/analysis.py`
- 图谱测试：`backend/graph_engine/tests.py`

扫描后确认当前已存在能力：

- 图谱页面已经支持基础展示、节点点击、边点击、图例和节点类型开关。
- 后端已经支持 Neo4j 优先读取，失败后自动回退到 MySQL。
- 后端已经提供合作网络概览、合作者类型分析、研究主题热点分析。
- 图谱接口已经向前端返回 `meta` 和 `analysis`。

扫描后确认当前断点：

- 还没有“路径分析一期”，用户难以理解某个节点为何与教师主体有关。
- 还没有轻量合作圈层概览，合作关系只能看热点，无法看圈层分布。
- 节点交互还缺少“从图谱节点回跳到成果证据页”的承接。
- 图谱来源说明、降级说明和交互边界说明还不够完整。

## 3. 本次落地范围

本次优先落地两类高价值增强，并同步补一部分交互与说明能力：

### 3.1 合作圈层概览

在 `backend/graph_engine/analysis.py` 中新增 `collaboration_circle_overview`，采用轻量阈值口径：

- 核心合作圈：同一合作者共同署名次数 `>= 3`
- 活跃合作圈：共同署名次数 `= 2`
- 扩展合作圈：共同署名次数 `= 1`

当前圈层分析只使用系统内真实论文与合作者关系数据，不做复杂社区发现。

### 3.2 路径分析一期

在 `frontend/src/views/AcademicGraph.vue` 中新增基于“当前已加载子图”的最短关系链说明：

- 关键词节点：优先解释为“教师 -> 论文 -> 关键词”
- 合作者节点：优先解释为“教师 -> 论文 -> 合作者”
- 成果节点：解释为与教师主体的直接关联证据

当前路径说明不是全库路径搜索，也不是复杂路径推理，只对当前页面已经加载出来的子图负责。

### 3.3 节点交互与筛选增强

图谱页补充了以下能力：

- 搜索节点名称或摘要
- 合作视角 / 主题视角 / 成果视角预设筛选
- 重置筛选
- 选中成果节点后回跳到成果页对应证据区

当前成果回跳只对教师视角开放，管理员不会获得越权进入成果维护入口的假联动。

### 3.4 图谱来源、降级与边界说明增强

在图谱接口 `meta` 中新增：

- `source_scope_note`
- `degradation_note`
- `interaction_note`

用于明确说明：

- 当前图谱覆盖了哪些节点范围
- Neo4j 不可用时会发生什么降级
- 当前路径说明与圈层说明的边界是什么

## 4. Neo4j 与 MySQL 双通道兼容策略

本次没有改变双通道主逻辑：

- 如果 `ENABLE_NEO4J=True` 且 Neo4j 可读，优先返回 Neo4j 图谱结果。
- 如果 Neo4j 不可用、驱动异常、查询失败，自动回退到 MySQL。
- 不管是哪条链路，前端都能获得统一的 `nodes + links + meta + analysis` 结构。

本次新增能力的兼容策略如下：

- 合作圈层概览基于系统内论文与合作者关系计算，不依赖复杂图引擎。
- 路径说明一期基于前端已加载子图做轻量 BFS，不依赖 Neo4j 算法扩展。
- 成果节点回跳依赖接口返回的 `entityId + recordType`，Neo4j 和 MySQL 两条链路都已补齐。

## 5. 当前已打通的增强链路

- 图谱节点搜索与视角筛选
- 图谱结果中的轻量合作圈层概览
- 图谱中关键词 / 合作者 / 成果节点的路径说明一期
- 图谱成果节点回跳到成果证据区
- 图谱来源范围说明、降级说明、交互边界说明

## 6. 当前仍明确保留的边界

- 当前没有做复杂最短路、全库路径检索或多路径排序。
- 当前没有做社区发现、子图挖掘或图机器学习。
- 当前合作圈层只是轻量阈值分层，不等同于正式圈层算法结果。
- 当前路径说明只对当前页面已加载子图负责，不代表系统全图谱的完整关系搜索。

## 7. 测试与验证

### 7.1 自动化验证命令

建议执行以下命令：

1. `cd frontend && npm run type-check`
2. `cd frontend && npm run verify:ui`
3. `cd backend && python manage.py check`
4. `cd backend && python manage.py test graph_engine --keepdb --noinput`

### 7.2 本次补充的测试点

后端新增或增强了以下断言：

- 图谱回退响应中包含新增的来源说明、降级说明、交互边界说明
- `collaboration_circle_overview` 返回结构完整
- 核心 / 活跃 / 扩展合作圈阈值划分可验证
- 成果类节点包含 `entityId` 与 `recordType`
- Neo4j 不可用时仍能经 MySQL 正常回退

前端 `verify:ui` 同步补充了以下检查：

- 图谱来源摘要中包含节点覆盖范围说明
- 图谱来源摘要中包含降级说明
- 图谱来源摘要中包含路径与交互边界说明

### 7.3 最小人工验证步骤

1. 用教师账号登录，进入画像页图谱区域。
2. 观察“数据来源”卡片，确认能看到来源范围、降级说明和交互边界说明。
3. 将视角切换到“合作视角”，点击某个合作者节点，确认“路径说明一期”出现。
4. 将视角切换到“主题视角”，点击某个关键词节点，确认“路径说明一期”出现。
5. 切换到“成果视角”，点击项目/知识产权/教学成果/学术服务节点，确认右侧可出现“查看支撑成果”入口。
6. 用管理员账号进入指定教师画像图谱区域，确认仍可查看图谱，但不会出现越权进入成果维护页的入口。

## 8. 后续建议

- 如果后续要继续增强，优先考虑“图谱报告摘要”而不是直接上重型图算法。
- 如果后续要做正式路径分析二期，建议仍先复用当前统一接口结构，在 `analysis` 中扩展结构化说明，不要直接引入复杂图平台依赖。
- 如果后续要增强合作圈层，可先增加更多可解释维度，例如近三年合作稳定度、校内外圈层占比，但仍建议保持轻量规则路线。
