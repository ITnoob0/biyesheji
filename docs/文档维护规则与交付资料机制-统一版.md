# 文档维护规则与交付资料机制-统一版
最后更新时间：2026-03-21

## 1. 目标

本文件用于定义当前仓库的文档维护机制，解决“过程文档很多，但现状不好找、边界不好讲、后续不好接”的问题。

这套机制服务于四类使用场景：

- 当前系统现状说明
- 后续开发交接
- 答辩演示与验收
- 论文写作与系统说明

## 3. 每轮功能变更后的强制同步项

### 3.1 接口清单

出现以下变化时，必须更新 [系统接口清单-统一版](e:/Project/BiShe/docs/%E7%B3%BB%E7%BB%9F%E6%8E%A5%E5%8F%A3%E6%B8%85%E5%8D%95-%E7%BB%9F%E4%B8%80%E7%89%88.md)：

- 新增接口或路由
- 删除接口
- 请求参数变化
- 返回结构变化
- 权限边界变化

### 3.2 模块边界

出现以下变化时，必须更新 [模块边界与系统最终形态说明-统一版](e:/Project/BiShe/docs/%E6%A8%A1%E5%9D%97%E8%BE%B9%E7%95%8C%E4%B8%8E%E7%B3%BB%E7%BB%9F%E6%9C%80%E7%BB%88%E5%BD%A2%E6%80%81%E8%AF%B4%E6%98%8E-%E7%BB%9F%E4%B8%80%E7%89%88.md)：

- 模块主职责变化
- 当前完成度变化
- 预留边界变化
- 主要工程支撑能力变化

### 3.3 数据口径

出现以下变化时，必须更新 [数据口径与风险记录-统一版](e:/Project/BiShe/docs/%E6%95%B0%E6%8D%AE%E5%8F%A3%E5%BE%84%E4%B8%8E%E9%A3%8E%E9%99%A9%E8%AE%B0%E5%BD%95-%E7%BB%9F%E4%B8%80%E7%89%88.md)：

- 新增资料字段或展示边界
- 新增统计指标
- 画像、推荐、问答、图谱、看板的解释口径变化
- 实时聚合 / 回退 / 离线聚合边界变化

### 3.4 风险记录

出现以下变化时，必须更新 [遗留问题记录](e:/Project/BiShe/docs/%E9%81%97%E7%95%99%E9%97%AE%E9%A2%98%E8%AE%B0%E5%BD%95.md) 或风险文档：

- 新增已知问题
- 新增环境依赖要求
- 新增启动、迁移、权限、回退风险
- 原有遗留问题已被解决

### 3.5 演示脚本

出现以下变化时，必须更新 [最终答辩演示步骤-第三轮](e:/Project/BiShe/docs/%E6%9C%80%E7%BB%88%E7%AD%94%E8%BE%A9%E6%BC%94%E7%A4%BA%E6%AD%A5%E9%AA%A4-%E7%AC%AC%E4%B8%89%E8%BD%AE.md)：

- 演示主链路变化
- 关键入口变化
- 答辩口径变化
- 演示前检查步骤变化

### 3.6 过程文档

每轮功能开发完成后，必须新增当轮过程文档，至少包含：

- 任务目标
- 修改范围
- 关键设计决策
- 验证命令与结果
- 已知边界与风险

## 4. 推荐的文档更新顺序

建议每轮开发按以下顺序同步：

1. 先完成代码与验证
2. 更新统一现状文档
3. 更新演示与验收文档
4. 补本轮过程文档
5. 最后回到 README 和索引检查入口是否失效

## 5. 已完成能力与预留边界的写法规则

文档中必须明确区分：

- 已完成：当前代码中已经可用、可验证、可演示的能力
- 预留边界：当前尚未落地，只是保留扩展空间的能力

必须避免以下写法：

- 把“计划中能力”写成“已上线能力”
- 把“轻量演示能力”写成“完整平台能力”
- 把“可选回退能力”写成“唯一主链路”

推荐使用的表述包括：

- “当前已支持”
- “当前仍属于轻量增强 / 规则增强 / 受控辅助”
- “当前尚未完成”
- “后续可在满足某些条件后继续推进”

## 6. 论文写作稳定口径

论文和系统说明建议优先引用以下文档口径：

- 系统整体形态与模块组成：模块边界文档
- 数据来源、统计口径与风险：数据口径文档
- 当前接口与交互边界：接口清单文档
- 完整版本方向：路线图文档

说明：
- 论文写作不应直接照搬过程文档中的阶段性措辞
- 若过程文档与统一现状文档不一致，以统一现状文档和代码为准
