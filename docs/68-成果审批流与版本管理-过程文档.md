# 68-成果审批流与版本管理-过程文档

## 目标

在现有科研管理系统中补齐成果审批流与版本管理能力，覆盖：

- 成果审批状态
- 审核通过 / 驳回
- 学院管理员按学院隔离待审核数据
- 已通过成果再次编辑后重新进入待审核
- 审核与字段变更时间轴

## 后端实现

### 模型层

更新文件：

- `backend/achievements/models.py`
- `backend/achievements/migrations/0009_achievement_review_flow_and_versioning.py`

主要改动：

- 为 `Paper / Project / IntellectualProperty / TeachingAchievement / AcademicService` 增加审批状态字段，状态枚举为：
  - `DRAFT`
  - `PENDING_REVIEW`
  - `APPROVED`
  - `REJECTED`
- `Project` 原有业务状态字段改为 `project_status`，避免与审批状态冲突，同时保留兼容兜底。
- 扩展 `AchievementOperationLog`：
  - 增加 `operator`
  - 增加 `change_details`
  - 增加 `review_comment`
  - 增加审批相关动作：`SUBMIT_REVIEW / APPROVE / REJECT`

### 序列化与接口层

更新文件：

- `backend/achievements/serializers.py`
- `backend/achievements/views.py`

主要改动：

- 各成果 Serializer 增加：
  - `status`
  - `status_label`
- `ProjectSerializer` 兼容旧请求中的 `status=ONGOING` 写法，并自动映射到 `project_status`
- 补充审批相关接口：
  - `POST /api/achievements/papers/<id>/submit-review/`
  - `POST /api/achievements/papers/<id>/approve/`
  - `POST /api/achievements/papers/<id>/reject/`
  - `GET /api/achievements/papers/pending-review/`
  - `GET /api/achievements/papers/<id>/workflow-logs/`
- 驳回接口强制要求 `reason`
- 审核日志写入：
  - 操作人
  - 变更字段
  - 字段前后值
  - 审核意见

### 权限边界

- 系统管理员：
  - 可查看全校待审核论文
  - 可审核通过 / 驳回
- 学院管理员：
  - 仅可查看本学院教师提交的 `PENDING_REVIEW` 论文
  - 仅可审核本学院教师论文
- 教师：
  - 仅可维护本人成果
  - 论文可主动“提交审核”
  - 已通过 / 已驳回成果再次编辑后会自动重置为 `PENDING_REVIEW`
