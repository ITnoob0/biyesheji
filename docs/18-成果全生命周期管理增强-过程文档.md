# 成果全生命周期管理增强-过程文档

## 1. 本次任务范围

本次改动基于现有成果模块增量开发，目标是在不破坏已有多类成果录入、基础编辑删除、搜索查看和 BibTeX 导入链路的前提下，把成果模块增强为更完整的成果管理中心。

本次明确保留：

- BibTeX 导入能力
- 导入前预览确认能力
- 多类成果录入与基础维护能力
- 画像主页的 MySQL 回退链路，不把 Neo4j 设为论文管理或画像展示强依赖

本次未做：

- 不引入外部高成本查重服务
- 不推翻现有成果模型与录入主流程
- 不新增复杂 RBAC 或成果审批流
- 不把预留能力表述为已完整上线能力

## 2. 仓库扫描结论

### 2.1 后端现状

- 成果模型集中在 `backend/achievements/models.py`
- 论文、项目、知识产权、教学成果、学术服务已具备基础 CRUD
- 论文当前已支持 DOI 去重、关键词抽取、合作者维护
- BibTeX 已有“预览 -> 确认导入”链路
- `dashboard-stats` 已可返回最近成果与总览统计

### 2.2 前端现状

- 成果中心主页面是 `frontend/src/views/AchievementEntryView.vue`
- 页面为“表单录入 + 基础列表”的单页模式
- 论文列表原先仅支持搜索和类型筛选
- BibTeX 弹窗已有预览与确认，但导入后异常反馈较轻
- 成果与画像、推荐、问答之间已有页面，但成果页缺少更清晰的联动入口

## 3. 本次落地方案

### 3.1 论文成果字段增强

在现有 `Paper` 模型上补充最小必要字段：

- `published_volume`
- `published_issue`
- `pages`
- `source_url`
- `is_representative`

设计原则：

- 不新建平行成果表
- 直接复用现有 `PaperSerializer` 和现有录入接口
- 允许字段为空，以兼容历史数据和旧录入链路

### 3.2 列表筛选与排序增强

论文列表新增：

- 按年份筛选
- 按代表作状态筛选
- 按时间、引用次数、题目、创建时间排序

实现方式：

- 后端在现有 `PaperViewSet.get_queryset()` 上增量加查询参数
- 前端复用现有查询状态 `queryMap.papers`

### 3.3 去重与重复提示策略

当前采用轻量去重策略，不引入复杂算法：

- 强校验：同一教师下 DOI 完全相同仍直接拦截
- 预警校验：同一教师下“题目 + 期刊/会议 + 年份”高度相似时给出重复提示
- BibTeX 预览阶段同时检查：
  - 当前账号已有 DOI 重复
  - 当前批次 DOI 重复
  - 当前账号已有题目/期刊/年份高度重复
  - 当前批次题目/期刊/年份高度重复

说明：

- “题目 + 期刊/会议 + 年份”仅作为轻量提示，不等价于严格查重结论
- 当前方案适合作为录入侧风控，不应对外宣称为完整论文查重系统

### 3.4 异常数据提示与导入反馈

本次对异常提示做了两层收口：

- 录入时：
  - 前端提示 DOI、页码、来源链接缺失的补充建议
  - 前端在重复风险场景下要求二次确认
- 导入时：
  - BibTeX 弹窗保留预览确认
  - 确认导入后展示成功、跳过、失败统计
  - 对跳过和失败记录展示首条摘要原因

### 3.5 代表作、最近成果与统计

新增或整合能力：

- 论文支持代表作标记
- 成果中心顶部复用 `dashboard-stats` 展示最近成果
- 新增论文 `summary` 接口，提供：
  - 代表作数量
  - 近三年论文数量
  - 缺 DOI 数量
  - 待补元数据数量
  - 按年份统计
  - 按类型统计
  - 最近论文记录

### 3.6 与画像、推荐、图谱、问答的联动

本次采用入口联动，不重做页面：

- 成果中心新增“画像与图谱主页”入口
- 新增“项目推荐入口”
- 新增“成果问答入口”
- 保留“个人中心”回跳入口

说明：

- 当前不新增独立图谱路由，继续复用画像主页中的图谱模块
- 当前联动是页面入口联动，不代表已实现“单条成果级智能推荐闭环”

## 4. 关键设计取舍

### 4.1 为什么只增强 `Paper`

论文是当前成果模块里最复杂、最需要字段完整度、最容易发生重复录入、同时与画像和推荐联动最紧密的类型。

因此本次优先把论文链路做完整，不对项目、知识产权、教学成果、学术服务做大规模结构改造，避免一次性改动过大。

### 4.2 为什么不做复杂查重

仓库当前没有为复杂查重准备独立索引、向量库或外部服务依赖。

本次如果强行引入复杂查重，会显著扩大改动范围并增加环境依赖，不符合“最小必要、可验证、可维护”的任务要求。

### 4.3 为什么复用画像主页承载图谱入口

当前路由中没有单独的图谱页面路由守卫配置；画像主页本身已经聚合画像和图谱展示能力，并保留 MySQL 回退链路。

因此本次采用“成果页跳到画像主页”的方式接入图谱入口，避免新增重复页面。

### 5.2 最小人工验证步骤

1. 登录教师账号并进入“教师成果录入中心”。
2. 检查顶部是否出现成果概览、最近成果和画像/推荐/问答入口。
3. 在论文表单中填写题目、摘要、时间、期刊、卷期页、来源链接，并勾选“代表作”。
4. 若 DOI 或题目/期刊/年份与现有记录重复，确认页面是否给出重复提示。
5. 保存后检查论文列表是否能看到：
   - 代表作标签
   - 元数据状态提示
   - 画像/图谱、推荐、问答入口
6. 使用年份、类型、代表作和排序条件筛选论文列表。
7. 点击 BibTeX 导入，先预览，再确认导入。
8. 若存在重复或异常记录，确认弹窗中能看到跳过或失败原因。
9. 点击“画像与图谱主页”“项目推荐入口”“成果问答入口”，确认都能进入现有受保护页面。

## 6. 当前边界与后续建议

当前边界：

- 代表作标记仅对论文开放
- 当前去重是轻量提示策略，不是全文查重系统
- 当前图谱联动采用入口复用，不新增独立成果图谱页
- 当前 BibTeX 仍以结构化字段导入为主，PDF 元数据解析继续保留为后续预留

后续建议：

- 在后端环境补齐 `rest_framework_simplejwt` 后补跑 Django 运行态测试
- 如后续继续增强，可优先考虑：
  - 单条成果详情抽屉或详情页
  - 历史版本与变更记录
  - 成果状态流转和审核日志
  - 更细粒度的批量导入校验报告导出
