# 高校教师科研画像与智能辅助系统：Codex Agent 高执行度提示词库

基于当前《优化版开发记录·现状校正版》整理，适用于后续分阶段开发、重构、补测、文档同步。

---

## 0. 使用原则

- 一次只给 Agent 一个明确任务，不要把多个大功能揉成一个提示词。
- 每个提示词都要包含：目标、现状约束、交付物、验收标准、禁止事项。
- 要求 Agent 先扫描仓库并确认影响范围，再动手改代码。
- 要求 Agent 优先保持当前主链路稳定：认证、教师信息、论文录入、画像展示、拓扑图展示。
- 每次改动后必须执行最小必要验证，并同步更新开发记录/API 文档/测试。

## 1. 通用母提示词（所有任务都建议先加）

```text
你现在是该项目的开发代理，请基于“当前已实现系统”继续增量开发，而不是推翻重做。

项目背景：
- 项目名称：高校教师科研画像与智能辅助系统
- 当前阶段版本：V1.1-portrait-baseline
- 当前一期主链路已经包含：认证与会话、教师基础信息、论文成果录入、教师画像展示、学术关系拓扑展示。
- 推荐系统、学院级看板、RAG/问答、复杂图分析不属于当前一期硬验收主链路。

必须遵守的硬约束：
1. 不得把认证状态从 sessionStorage 改为 localStorage。
2. 教师登录账号必须继续使用工号，不能回退为 teacher1 / teacher2 一类用户名。
3. Neo4j 不能成为论文录入和画像展示的强依赖；若不可用，系统必须能走 MySQL 回退链路。
4. 所有新增页面都必须接入现有严格路由守卫。
5. 所有新增功能都要注明：是否纳入当前阶段验收。
6. 新增功能时，优先考虑职责拆分；如果一个文件同时包含页面渲染、表单状态、数据库读写、业务规则和工具函数，应拆成多个模块
7. 修改接口、数据结构、页面逻辑后，必须同步补充测试或至少补充可复现验证步骤。
8. 不要假设推荐相关表、项目指南表、画像快照表已经存在；必须以当前真实模型为起点增量设计。

你的工作方式：
- 第一步：先扫描仓库，识别相关前后端目录、现有数据模型、接口、页面、测试位置，并给出“将修改哪些文件”的清单。
- 第二步：给出最小可行实现方案，优先复用现有模式与组件，不做无关重构。
- 第三步：开始实现。
- 第四步：实现后输出：
  a) 修改文件列表
  b) 核心改动说明
  c) 数据库变更/迁移说明
  d) 测试或验证步骤
  e) 风险与后续建议

输出要求：
- 不要只给建议，直接产出可合并代码。
- 若需求过大，请主动拆成 2~4 个可连续执行的子任务，并先完成第 1 个子任务。
- 若发现当前仓库实现与需求冲突，优先保持主链路稳定，并明确说明取舍。
```

```text
基于当前项目，继续做“一期成果管理闭环增强”，目标是：在不破坏现有论文主工作流的前提下，为以下成果类型补齐最小可用前后端闭环：

1. 盘点现有模型、序列化器、视图、路由、前端页面、类型定义。
2. 为上述 4 类成果补齐最小必要接口（列表、创建；如成本可控再补删除/编辑）。

- 原论文功能不受影响。

请按以下格式工作：
1. 先扫描并确认现有成果模块代码结构。
2. 列出计划修改文件。
3. 实现代码。
4. 输出数据库迁移说明。
5. 输出验证步骤。

特别禁止：
- 不要重写认证系统。
- 不要引入新的状态管理体系。
- 不要把这 4 类成果强行抽象成一个大而全的通用动态表单引擎。
```

## 3. 提示词：让画像页联动多成果类型

```text
请在当前项目中继续增强“教师科研画像首页”，让画像页不再只主要依赖论文数据，而是能够联动多成果类型（项目、知识产权、教学成果、学术服务）。

- 当前画像页已经有基础信息、统计卡片、雷达图、趋势图、关键词画像、合作画像、拓扑图。
- 当前画像页部分数据由后端提供，部分由前端基于论文列表二次计算。
- 现阶段不能引入大规模画像重算任务系统。

任务目标：
1. 梳理当前画像页的数据来源与计算方式。
2. 在不推翻现有结构的情况下，把多成果类型纳入统计与展示。
3. 优先增强以下内容：
   - 统计卡片
   - 能力维度来源说明
   - 最近成果列表或代表性成果区
   - 页面中的“数据更新时间/数据来源说明”

- 能复用现有 dashboard-stats / radar 接口模式就复用。
- 能在前端轻量聚合的，不必强行后端化。

验收标准：
- 画像页能体现多成果类型，不再仅从论文视角理解教师能力。
- 原有论文趋势、关键词、合作画像仍保持可用。
- 管理员查看指定教师画像的能力不被破坏。
- 无 Neo4j 情况下仍可正常展示页面主体。

请输出：
- 当前画像数据流梳理
- 改造方案
- 代码实现
- 影响到的接口清单
- 验证步骤
```

## 5. 提示词：设计并落地 BibTeX / PDF 批量导入的一期方案

```text
请为当前项目新增“成果批量导入”的一期最小可用方案，优先考虑 BibTeX，其次再评估 PDF 元数据解析；目标不是一次性做全自动智能解析，而是做出可演示、可验收、低风险的一期能力。

项目现状：
- 当前论文录入链路已打通。
- AI 标签提取、异步任务、复杂导入尚未纳入一期硬验收。
- 当前系统必须优先保证手工录入可用。

你的任务：
1. 先分析当前 Paper、Keyword、CoAuthor 写入链路，确定复用点。
2. 设计一个最小导入流程，例如：上传 BibTeX 文件 -> 后端解析 -> 预览 -> 用户确认导入。
3. 如 PDF 导入成本过高，请先只实现 BibTeX，并把 PDF 解析作为预留接口或占位设计。
4. 导入失败不能影响既有论文数据。
5. 如需异步化，请仅在确有必要时引入轻量方案；不要把任务系统做得过重。

交付要求：
- 新增导入接口与必要页面/弹窗
- 给出错误处理与重复 DOI 去重策略
- 保持与现有论文手工录入逻辑一致
- 补充验证步骤

验收标准：
- 用户可上传 BibTeX 文件并导入论文数据。
- 重复 DOI 或关键字段异常有明确提示。
- 导入后的论文能正常进入现有画像统计链路。
```

## 11. 最推荐的实际使用顺序

1. 先用“侦察版提示词”让 agent 扫仓库。
2. 再用“补齐多成果类型闭环”。
3. 然后做“画像页联动多成果类型”。
4. 再做“批量导入”。
5. 之后做“项目指南管理与推荐一期”。
6. 最后做“管理端统计看板”与“工程收口”。

这样更符合当前项目状态，也更稳妥。
