# 测试、验证与工程化增强 - 过程文档

## 1. 任务目标

本轮任务是在现有后端关键测试、前端 `type-check` 和最小 UI 验证脚本基础上，继续增强测试基线与验证体系，让开发 16 到 24 对应的关键链路具备更稳定的持续开发基础。

本次重点覆盖：

- 权限边界
- 成果导入与编辑回归
- 推荐结果与管理员分析
- 问答失败回退
- 图谱 MySQL 回退链路
- 统一验证命令与演示脚本

## 2. 扫描结果

### 2.1 前端测试设施

当前前端已有：

- `frontend/package.json`
  - `type-check`
  - `verify:ui`
  - `build`
- `frontend/scripts/verify-third-round-ui.mjs`

说明：

- 当前前端没有引入 Vitest / Playwright 一类更重的测试框架
- 现有基线主要依赖静态类型检查和基于 helper 的准自动化验证

### 2.2 后端测试设施

当前后端已有较完整的 `APITestCase`：

- `backend/users/tests.py`
- `backend/achievements/tests.py`
- `backend/graph_engine/tests.py`
- `backend/project_guides/tests.py`
- `backend/ai_assistant/tests.py`

说明：

- 这些测试已覆盖大部分核心接口
- 但验证入口分散，缺少一个面向演示和回归的统一脚本

### 2.3 CI 与工程化现状

扫描结果：

- 当前仓库未发现 `.github` 工作流目录
- 当前没有现成的 GitHub Actions / CI 文件

因此本次工程化增强重点放在：

- 可重复执行的本地验证命令
- 统一回归脚本
- 更明确的验证文档

## 3. 本次落地内容

### 3.1 前端测试基线增强

更新文件：

- `frontend/package.json`

新增脚本：

- `verify:baseline`
  - 串联 `type-check` + `verify:ui`
- `verify:full`
  - 串联 `verify:baseline` + `build-only`

价值：

- 将原本分散的前端验证命令收口为更清晰的基线入口
- 便于答辩演示前、提交前和回归时重复执行

### 3.2 权限边界测试增强

更新文件：

- `backend/users/tests.py`
- `backend/achievements/tests.py`

新增或增强场景：

- 教师不能修改其他教师账户详情
- 教师不能编辑或删除其他教师论文
- 教师访问管理员看板仍保持 403

### 3.3 成果导入与编辑回归测试增强

更新文件：

- `backend/achievements/tests.py`

新增或增强场景：

- BibTeX 预览识别“当前批次内 DOI 重复”
- BibTeX 确认导入会跳过重复 DOI，而不是错误写入
- 论文越权更新 / 删除被拒绝
- 顺手清理了一段历史遗留的测试乱码文案，避免误导后续定位

### 3.4 推荐、问答、图谱回退测试增强

更新文件：

- `backend/project_guides/tests.py`
- `backend/ai_assistant/tests.py`
- `backend/graph_engine/tests.py`

新增或增强场景：

- 推荐管理员分析字段分布回归
- 双教师对比时 `comparison_summary` / `admin_analysis` 保持返回
- 问答失败回退结果包含失败原因与边界说明
- Neo4j 可用开关开启但运行时不可用时，图谱仍回退到 MySQL 并保持轻量分析结果

### 3.5 统一演示与回归脚本

新增文件：

- `backend/scripts/verify_development_16_24_regression.py`

作用：

- 自动执行 `check`
- 自动刷新演示账号 `init_demo_teachers`
- 用 Django `APIClient` 依次验证管理员和教师的关键登录后主链路

覆盖的关键接口包括：

- 登录
- 当前用户信息
- 画像统计
- 雷达
- 成果列表
- 图谱
- 推荐
- 问答
- 学院看板权限边界

这类脚本比单纯跑单元测试更适合做：

- 答辩前自检
- 演示前联调检查
- 多轮功能增量后的主链路回归

## 4. 本次验证命令

### 4.1 实际执行

- `npm run verify:baseline`
- `python manage.py check`
- `python manage.py test --keepdb --noinput users achievements graph_engine project_guides ai_assistant`
- `python scripts/verify_development_16_24_regression.py`

### 4.2 实际结果

- 前端基线验证通过
- Django `check` 通过
- 后端 61 个测试通过
- 开发 16 到 24 统一回归脚本通过

## 5. 最小人工演示步骤

### 5.1 教师链路

1. 教师账号登录。
2. 打开个人中心，确认资料与账户安全区可见。
3. 打开成果中心，验证录入、筛选与 BibTeX 导入入口。
4. 打开画像主页，确认画像趋势、结构与图谱可加载。
5. 打开推荐页与问答页，确认推荐与问答结果正常返回。
6. 尝试进入学院看板，确认被拒绝并回到受保护页。

### 5.2 管理员链路

1. 管理员登录。
2. 进入教师管理，确认教师列表与密码重置链路。
3. 进入项目指南管理，确认管理入口正常。
4. 进入学院看板，确认筛选与统计可加载。
5. 查看指定教师画像、推荐与问答，确认管理员视角可切换。

## 6. 历史遗留问题与本次新增问题区分

### 6.1 历史遗留问题

- 当前仓库没有现成 CI 工作流
- 本地测试库 `test_biyesheji` 已存在，直接执行 `manage.py test` 可能触发交互确认

### 6.2 本次处理结果

- 本次没有引入新的测试框架重构
- 本次通过 `--keepdb --noinput` 收口了后端测试执行方式
- 本次新增了统一回归脚本和更系统的验证文档

## 7. 后续建议

建议后续继续做三件事：

1. 若项目进入更稳定阶段，可再考虑引入轻量 CI，将 `manage.py check`、关键后端测试和 `npm run verify:baseline` 自动化。
2. 若前端交互继续复杂化，可再评估是否引入组件级测试框架，但不建议在当前阶段一次性上重型 E2E。
3. 后续每轮增量开发都优先把新主链路并入 `verify_development_16_24_regression.py` 这类统一脚本，而不是只追加零散文档说明。
