# 学术社交拓扑图交互增强-过程文档

## 1. 任务背景

本轮任务目标是在不破坏当前画像主页稳定性的前提下，增强“学术社交拓扑图”模块的交互体验。  
硬约束如下：

- Neo4j 不能成为强依赖
- MySQL 回退链路必须保持可用
- topology 接口需尽量保持向后兼容
- 图谱异常不能拖垮整个画像页

本次改动纳入当前阶段验收，因为它属于教师画像主页主链路的增强项。

## 2. 改造前链路梳理

### 2.1 后端

- 图谱接口位置：`backend/graph_engine/views.py`
- 接口路径：`GET /api/graph/topology/<user_id>/`
- 现有逻辑：
  - 先校验用户访问权限
  - 优先尝试 Neo4j 图谱查询
  - 若 Neo4j 关闭、不可用或无结果，则回退到 MySQL 关系数据组装

### 2.2 前端

- 图谱组件位置：`frontend/src/views/AcademicGraph.vue`
- 页面调用方式：
  - 在画像页中作为子模块展示
  - 读取当前画像教师 `userId`
  - 请求 `/api/graph/topology/<userId>/`
  - 使用 ECharts `graph` 系列渲染

### 2.3 已有能力

- 节点基础点击
- 基础颜色区分
- 基础空态文案

### 2.4 主要不足

- 缺少图谱数据来源说明，用户不知道当前是 Neo4j 还是 MySQL 回退
- 没有图例筛选，节点多时不够易读
- 只对节点信息展示较弱，边关系解释不清晰
- 请求失败时提示不够友好
- 空数据态没有明确引导应先补录哪些成果

## 3. 设计取舍

本次没有引入新的图谱快照体系，也没有把图谱逻辑重写为 Neo4j 专属能力。  
采用的策略是：

- 后端在原有 `nodes`、`links` 结构上做兼容式增强
- 前端只依赖公共字段实现交互，不写只适用于 Neo4j 的逻辑
- 对图谱异常做局部兜底，不影响画像页其他模块

## 4. 实现内容

### 4.1 后端增强

修改文件：`backend/graph_engine/views.py`

增强点：

- 保留旧字段：
  - `nodes`
  - `links`
- 兼容式新增：
  - `meta.source`
  - `meta.fallback_used`
  - `meta.node_count`
  - `meta.link_count`
  - `meta.notice`
- 为节点新增辅助展示字段：
  - `nodeTypeLabel`
  - `detailLines`
- 为边新增辅助展示字段：
  - `relationLabel`
  - `description`

这样做的原因：

- 不破坏原有前端依赖
- 前端可以基于新增字段增强交互，而旧逻辑仍可工作
- 可显式告知当前图谱来源与回退情况

### 4.2 前端增强

修改文件：`frontend/src/views/AcademicGraph.vue`

本次增强的交互项：

- 节点点击查看详情
- 边点击查看关系说明
- 节点类型图例与筛选
- 数据来源与回退状态提示
- 更友好的加载失败提示
- 更明确的空数据引导

页面行为说明：

- Neo4j 可用时显示“Neo4j 实时图谱”
- Neo4j 不可用时显示“MySQL 回退图谱”
- 失败时仅当前图谱区域展示错误态，画像页其他内容继续可用
- 过滤逻辑仅基于公共节点类型字段，不依赖 Neo4j 专属数据

### 4.3 测试补充

修改文件：`backend/graph_engine/tests.py`

补充内容：

- 验证 MySQL 回退响应中保留 `nodes`、`links`
- 验证新增 `meta` 字段存在且值正确
- 验证节点详情字段与边关系说明字段存在
- 验证空数据时仍保持兼容响应结构

## 5. 兼容性说明

本轮 topology 接口为兼容式扩展：

- 旧前端若只读取 `nodes`、`links`，不会受影响
- 新前端使用新增字段增强交互
- Neo4j 与 MySQL 回退两条链路都输出统一的扩展字段

因此不存在“只在 Neo4j 场景可运行”的前端逻辑。
