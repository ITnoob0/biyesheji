# 文档与交付资料统一 - 过程文档

## 1. 任务背景

当前仓库已经积累了多轮过程文档、验收文档、演示文档和遗留问题记录。  
这些文档能反映开发过程，但存在以下问题：

- 现状说明分散
- 接口清单未统一
- 模块边界说明分散在多轮过程文档中
- 数据口径与风险边界需要统一口径
- 缺少一个面向开发交接、论文写作和答辩讲解的统一索引

## 3. 本次整理策略

本次没有删除历史文档，也没有重写全部旧文档，而是采用：

- 保留历史过程文档
- 新增统一总览与统一说明文档
- 对现有验收 / 演示 / 遗留文档补统一入口链接

这样做的原因：

- 可保留阶段演进痕迹
- 可避免大规模文档重写
- 可让后续读者快速分辨“历史阶段描述”和“当前统一状态”

## 5. 与代码一致性的处理方式

本次统一文档尽量只写当前代码已经存在的内容，主要依据：

- 后端真实 URL 配置
- 前端真实路由与导航
- 已经通过的验证命令
- 已经落库的功能字段与权限边界

明确避免了以下风险：

- 不把未来 RAG 写成当前已完成
- 不把规则增强推荐写成模型推荐已完成
- 不把图分析写成复杂图挖掘平台
- 不把轻量学院看板写成完整 BI 平台
- 不把画像趋势写成完整历史快照体系

## 7. 后续建议

后续如果继续增量开发，建议遵循以下文档维护规则：

1. 新功能过程继续写单独过程文档
2. 涉及接口、模块边界、数据口径变更时，同时更新统一文档
3. 涉及答辩主链路变动时，同时更新演示步骤和验收清单
4. 不再把“当前状态”只埋在单轮过程文档里
