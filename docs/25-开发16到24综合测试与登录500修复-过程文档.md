# 开发 16 到 24 综合测试与登录 500 修复 - 过程文档

## 1. 背景

用户反馈：前端登录后页面出现“飘红乱码”。  
实际截图显示：

- 浏览器地址仍停留在登录页
- 控制台报错为 `POST /api/token/ 500 (Internal Server Error)`
- 页面左侧出现大量红色 HTML 文本

这说明问题并非单纯的前端编码错误，而是登录接口返回了 Django Debug 调试页，前端把原始错误内容暴露到了界面层。

## 3. 根因定位

### 3.1 第一层问题：后端缺失依赖

此前后端 `manage.py check` 无法启动，原因是环境缺少：

- `djangorestframework-simplejwt`

这导致 Django 启动和测试命令都受阻，也让综合测试无法真正执行。

### 3.2 第二层问题：数据库迁移未应用

依赖补齐后检查迁移状态，发现以下迁移尚未落库：

- `users.0002_customuser_password_lifecycle`
- `users.0003_customuser_personal_center_fields`
- `achievements.0005_paper_lifecycle_fields`
- `project_guides.0002_projectguide_rule_profile_and_tags`

而登录后的账户链路已经会读取如下字段：

- `password_reset_required`
- `password_updated_at`
- `avatar_url`
- `contact_phone`

因此数据库结构与代码不一致时，`/api/token/` 登录链路会在用户查询或后续 `me` 接口读取时触发后端 500。  
用户截图中的红色“乱码”本质上是 Django Debug HTML 被前端原样展示。

### 3.3 第三层体验问题：前端错误提示缺乏 HTML 调试页兜底

前端错误提示工具此前会直接采用后端返回的字符串内容。  
如果后端返回的是整段 HTML traceback，前端可能把调试页文本直接显示给用户，造成“飘红乱码”的观感。

## 6. 关于“飘红乱码”的结论

本次问题的主因不是前端中文编码损坏。  
根因链路如下：

1. 开发 16 到 24 引入的新字段已经进入代码读取路径
2. 但数据库未执行对应迁移
3. 登录链路请求触发后端 500
4. Django 在 `DEBUG=True` 下返回 HTML 调试页
5. 前端把该错误内容暴露出来，形成“飘红乱码”

因此本次修复分为两层：

- 根修：补依赖、补迁移、恢复接口
- 兜底：前端不再展示原始 HTML 调试页

## 7. 当前边界

本次已修复登录 500 和错误展示问题，但仍保留以下开发期特征：

- Django 仍处于 `DEBUG=True`
- Vite 构建仍存在较大的 chunk warning
- 测试依赖历史保留的 `test_biyesheji` 测试库，因此本次使用 `--keepdb --noinput`

这些不影响当前登录与核心链路正确性，但属于后续可以继续治理的工程项。

## 8. 后续建议

建议后续继续做三件事：

1. 补一份后端依赖清单文件，如 `requirements.txt` 或 `pyproject.toml`
2. 为部署/联调环境补一条启动前检查：未执行迁移时直接提示
3. 后续将登录与全局 API 错误提示统一接入 HTML 调试页防泄漏规则
