## 一、推荐的答辩演示主路径

建议最终答辩按照“主链路 + 扩展亮点”方式展开。

### 演示前准备步骤

建议在正式演示前先按下面顺序做一次启动前检查：

1. 进入 `backend`
2. 执行 `python scripts\startup_preflight.py`
3. 若提示存在未执行迁移，则执行 `python manage.py migrate`
4. 再次执行 `python scripts\startup_preflight.py`
5. 执行 `python scripts\restore_demo_state.py`
6. 最后启动后端与前端服务

这样可以提前避免“代码已读新字段，但数据库未迁移”这类问题影响答辩或演示。

如需在正式答辩前额外确认核心业务闭环，可继续执行：

- `python scripts\verify_development_16_24_regression.py`

它会对当前最关键的教师 / 管理员主链路做一轮准自动化回归验证。

### 第一部分：主链路

1. 使用教师工号登录
2. 进入教师画像主页
3. 查看教师基础资料与多成果画像
4. 进入基础档案页修改信息
5. 进入成果页新增或编辑一条成果
6. 返回画像页观察画像与图谱联动变化

### 第二部分：扩展亮点

7. 进入项目推荐页，展示推荐结果、标签分组和推荐说明
8. 点击“智能解读”，进入问答页说明推荐原因
9. 返回首页或看板，进入学院级统计看板
10. 演示学院 / 年份 / 教师筛选
11. 从排行钻取到教师画像或推荐页
12. 导出当前排行 CSV
13. 回到图谱页，展示合作热点与主题热点分析

## 二、演示时建议强调的口径

### 主链路口径

- 系统核心能力围绕教师科研画像主链路展开
- 以教师真实资料、成果数据和关系数据为基础
- 不依赖 Neo4j 才能完成主展示

### 扩展能力口径

- 推荐系统：规则增强型、可解释推荐
- 学院看板：轻量管理分析看板
- 智能问答：轻量单场景演示
- 图分析：轻量图分析亮点

## 三、演示中的风险规避建议

- 若 Neo4j 不可用，提前说明系统仍会回退到 MySQL 关系数据
- 若推荐结果较少，可提前准备一位成果较完整的教师账号
- 若问答结果需更稳定，优先演示“画像总结”与“推荐原因说明”
- 学院看板演示时优先选择已有成果数据的学院和年份

## 五、结论

第三轮后的推荐演示路径应当突出：

- 主链路稳定

- 扩展能力可解释

这样更容易在答辩中同时体现系统完成度和工程克制性。
