# 29-最小 CI 与自动化验证流水线-过程文档

## 1. 本轮目标

在当前系统已经具备前端基线验证、后端关键测试、统一主链路回归脚本和启动前检查的基础上，补一套最小可用的自动化验证流水线，减少“靠人工记忆执行命令”的风险。

本轮重点不是重构工程体系，而是把已有真实命令标准化、可重复化、可接入 CI 化。

## 2. 扫描结论

### 2.1 前端真实命令

来自 `frontend/package.json`：

- `npm run type-check`
- `npm run verify:ui`
- `npm run verify:baseline`
- `npm run verify:full`

其中 `verify:baseline` 已经能覆盖：

- `type-check`
- `verify:ui`

因此本轮优先复用 `verify:baseline` 作为前端最小基线。

### 2.2 后端真实命令

当前仓库已存在的关键后端验证入口：

- `python scripts/startup_preflight.py`
- `python manage.py check`
- `python manage.py test --keepdb --noinput users achievements graph_engine project_guides ai_assistant`
- `python scripts/verify_development_16_24_regression.py`

### 2.3 CI 现状

扫描时仓库中不存在 `.github/workflows`，说明此前没有正式 GitHub Actions 配置。

### 2.4 依赖清单现状

扫描时前端存在：

- `frontend/package-lock.json`

扫描时后端不存在正式 `requirements.txt` 或 `pyproject.toml`，这会导致 CI 难以直接安装与当前环境一致的后端依赖。

## 3. 本轮改动

### 3.1 新增后端依赖清单

新增：

- `backend/requirements.txt`

作用：

- 为本地环境初始化提供统一入口
- 为 GitHub Actions 提供稳定的后端安装清单

本轮保持克制，只写入当前后端运行和验证必需的依赖，不把 Neo4j Python 驱动写成强依赖。

### 3.2 新增统一自动验证入口

新增：

- `backend/scripts/run_validation_pipeline.py`

该脚本不重新发明命令体系，而是顺序调用已有验证入口：

1. `python scripts/startup_preflight.py`
2. `python manage.py check`
3. `python manage.py test --keepdb --noinput users achievements graph_engine project_guides ai_assistant`
4. `python scripts/verify_development_16_24_regression.py`
5. `npm run verify:baseline`

输出策略：

- 每个步骤开始前输出 `[STEP]`
- 成功输出 `[PASS]`
- 失败输出 `[FAIL]` 和最小处理建议

这样既能作为本地统一回归入口，也能直接作为 CI 的主执行命令。

### 3.3 新增最小 GitHub Actions 工作流

新增：

- `.github/workflows/validation.yml`

工作流设计要点：

- 使用 `frontend/package-lock.json` 执行 `npm ci`
- 使用 `backend/requirements.txt` 安装 Python 依赖
- 启动 MySQL 8 服务
- 先执行 `python manage.py migrate`
- 再执行统一自动验证入口 `python backend/scripts/run_validation_pipeline.py`

### 3.4 文档更新

更新：

- `README.md`

补充内容：

- 后端依赖安装步骤
- 本地验证顺序
- 统一自动验证入口说明
- CI 验证范围说明

## 4. 设计权衡

### 4.1 为什么没有直接做大而全 CI

当前系统的目标是先建立最小可用闭环，而不是一次性覆盖所有页面、所有命令和所有环境组合。

因此本轮只保留：

- 前端基线验证
- 后端检查
- 后端关键测试
- 统一主链路回归

这是当前最能防止高风险回归的一组组合。

### 4.2 为什么新增统一脚本而不是只写 workflow

如果只有 GitHub Actions YAML，本地开发仍然要靠人工记忆命令。

新增 `backend/scripts/run_validation_pipeline.py` 的好处是：

- 本地和 CI 复用同一套执行顺序
- 失败日志更集中
- 后续继续扩展验证范围时，只需要维护一处主入口

### 4.3 为什么没有把 Neo4j 变成 CI 强依赖

系统当前明确要求：

- Neo4j 是可选能力
- MySQL 回退链路必须保留

因此本轮 CI 明确在 `ENABLE_NEO4J=0` 下执行最小验证，确保图谱和画像主链路不会因为 Neo4j 缺失而阻塞交付。

## 5. 历史遗留问题与本轮边界

### 5.1 历史遗留问题

- 仓库此前没有正式 GitHub Actions 工作流
- 后端此前缺少正式依赖清单
- 本地验证顺序主要依赖 README 和记忆

### 5.2 本轮未做

- 没有引入完整矩阵 CI
- 没有接入前端 E2E 自动化
- 没有接入部署流水线
- 没有把所有模块测试都并入更大规模覆盖

这些都属于后续增强方向，不是本轮最小闭环的范围。

## 6. 本轮验证

计划验证命令：

- `cmd /c npm run type-check`
- `cmd /c npm run verify:baseline`
- `python manage.py check`
- `python manage.py test --keepdb --noinput users achievements graph_engine project_guides ai_assistant`
- `python scripts/verify_development_16_24_regression.py`
- `python scripts/run_validation_pipeline.py`

若 GitHub Actions 工作流本轮未在远端平台实际执行，应在最终汇报中明确说明“配置已落地，但未在 GitHub 平台实际触发”。
