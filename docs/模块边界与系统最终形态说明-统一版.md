# 模块边界与系统最终形态说明-统一版

最后更新时间：2026-03-24

## 1. 系统总体结构

当前系统可以理解为一个围绕“教师科研画像主链路”组织的多模块平台。  
主要模块包括：

- 认证与账户
- 教师个人中心
- 成果管理中心
- 教师画像
- 学术图谱与图分析
- 项目推荐与指南管理
- 学院级看板
- 智能问答辅助
- 测试验证与工程化支撑

系统最终形态的核心不是单一页面，而是：

- 以教师画像为分析主视图
- 以成果中心为数据维护主入口
- 以推荐、问答、图谱和看板作为扩展分析能力

## 2. 模块边界说明

### 2.1 认证与账户

当前完成程度：

- 工号登录
- 教师注册
- 忘记密码
- 当前用户资料维护
- 管理员教师账户管理
- 管理员教师账户启停 / 恢复
- 管理员重置密码
- 管理员批量账户操作
- 教师主动修改密码
- 密码生命周期提示与恢复说明
- 严格路由守卫
- 统一权限能力入口 `permission_scope`

当前边界：

- 登录态继续使用 `sessionStorage`
- 教师主认证方式继续是工号
- 当前仍是教师 / 管理员的轻量角色边界，不是复杂 RBAC
- 当前多角色扩展的正式保留入口是：
  - `role_code`
  - `role_label`
  - `permission_scope`
  这三者当前用于双角色统一收口，不代表复杂多角色平台已经完整上线
- 教师自助范围当前主要包括：
  - 维护自己的个人资料和密码
  - 维护自己的成果
  - 查看自己的画像、推荐、图谱和问答结果
- 管理员范围当前主要包括：
  - 管理教师账户和重置教师密码
  - 批量停用、恢复和批量重置教师账户
  - 查看指定教师画像、推荐、图谱、问答和学院级统计
  - 维护项目指南
- 当前管理员仍不通过教师自助成果入口代替教师录入、修改或删除成果
- 教师管理入口当前仅面向教师账户，不把管理员账户当作“教师资料”处理
- 当前管理员账户操作审计仍是轻量日志级别，不是独立审计平台

后续建议：

- 若后续需要多角色扩展，可先在现有 `role_code / role_label + permission_scope` 基础上做增量扩展
- 若后续需要更细角色治理，应继续优先沿用统一权限入口与统一提示，不建议直接重构成复杂 RBAC 平台
- 不建议直接重写认证体系

### 2.2 教师个人中心

当前完成程度：

- 已从基础档案页升级为个人中心
- 支持教师资料、账户安全、代表性成果和统一快捷入口
- 支持头像上传与头像 URL 备用方式
- 支持联系方式展示策略
- 教师默认工作台入口已优先收口到个人中心
- 已与成果、推荐、问答形成入口联动

当前边界：

- 当前更像“系统内个人中心”，不是独立公开主页系统
- 当前头像上传是本地媒体文件级能力，不是完整素材管理系统
- 当前公开字段 / 内部字段边界已经明确，但字段级细粒度权限控制仍保持克制
- 当前联系方式策略主要控制邮箱 / 电话在系统内资料摘要中的展示方式，不代表匿名公网可见性体系

后续建议：

- 如后续需要增强，优先考虑更多资料可见性细分和教师侧工作台摘要深化

### 2.3 成果管理中心

当前完成程度：

- 支持论文、项目、知识产权、教学成果、学术服务
- 支持新增、编辑、删除、搜索、筛选
- 论文支持 BibTeX 预览确认导入
- 支持轻量重复提示、代表作、统计摘要

当前边界：

- 当前去重仍是轻量规则，不是复杂查重引擎
- 当前没有成果审批流或版本历史

后续建议：

- 如后续增强，优先补版本记录或导入纠错体验

### 2.4 教师画像

当前完成程度：

- 画像主页
- 雷达评分
- 维度解释
- 趋势展示
- 近年成果结构
- 关键词演化与主题聚焦
- 画像形成说明

当前边界：

- 当前趋势主要基于成果年份回溯估算
- 当前未引入真正的历史画像快照体系

后续建议：

- 若需要做画像快照，必须先定义冻结口径与版本语义

### 2.5 学术图谱与图分析

当前完成程度：

- 学术图谱展示
- Neo4j 优先读取
- MySQL 回退链路
- 合作网络概览
- 合作者类型分析
- 研究主题热点分析

当前边界：

- 当前图分析是轻量展示增强
- 还不是复杂路径分析、圈层分析、社区发现平台

后续建议：

- 后续如继续增强，优先做路径分析或合作圈层，而不是一次性做复杂图挖掘

### 2.6 推荐系统与项目指南

当前完成程度：

- 项目指南管理
- 规则增强型推荐
- 推荐标签
- 推荐解释维度
- 管理员教师对比
- 管理员分析摘要

当前边界：

- 当前仍是规则增强路线
- 不是复杂机器学习推荐系统

后续建议：

- 后续先补用户反馈信号，再评估是否进入更强模型化推荐

### 2.7 学院级看板

当前完成程度：

- 基础统计
- 筛选
- 趋势分析
- 对比摘要
- 教师排行
- 钻取到画像与推荐

当前边界：

- 当前是轻量学院级管理分析入口
- 还不是完整 BI 平台

后续建议：

- 后续可继续补细粒度钻取，但不建议大规模 BI 化重构

### 2.8 智能问答辅助

当前完成程度：

- 基于真实系统数据的模板化问答
- 支持画像、成果、推荐、学院统计说明
- 支持来源说明、边界提示和失败回退

当前边界：

- 当前不是通用知识平台
- 当前没有 RAG、向量检索或外部知识库

后续建议：

- 若未来进入 RAG，必须先确定知识来源、索引策略和更新机制

### 2.9 测试验证与工程化

当前完成程度：

- 前端 `type-check`
- 前端 `verify:ui`
- 前端 `verify:baseline`
- 后端关键测试
- 开发 16 到 24 统一回归脚本

当前边界：

- 当前已经具备最小 CI 与自动化验证流水线，覆盖 `manage.py check`、关键测试、前端基线验证和统一主链路回归
- 当前前端仍没有完整组件级测试框架

后续建议：

- 已先补最小 CI；后续再根据真实收益决定是否引入更重的前端测试体系、更多环境矩阵或更细粒度回归

## 3. 系统最终形态说明

从当前代码和文档状态看，系统最终形态更适合描述为：

- 一个围绕教师科研画像构建的数据整合、分析展示与轻量智能辅助平台

最终形态中已经具备雏形的关键特征包括：

- 资料、成果、画像、图谱、推荐、问答之间的联动
- 教师与管理员两个清晰视角
- 以真实系统数据驱动的解释性能力

## 4. 当前仍未完成的系统级边界

- 复杂多角色与完整 RBAC
- 公开个人主页体系
- 高级图挖掘平台
- 机器学习推荐平台
- 外部知识驱动的完整问答平台
- BI 级离线聚合与多层报表

这些方向当前都属于“可继续演进的完整版本方向”，不应表述为已完整上线。

## 5. 维护说明

当以下内容变化时，应同步更新本文档：

- 某个模块的当前完成度变化
- 某个模块的边界收缩或扩展
- 工程支撑能力发生明显变化
- 系统最终形态口径发生调整

本文档用于描述“当前系统已经做到什么”和“当前仍未做到什么”，不应把预留能力写成已完整上线能力。
