# 第二轮开发

最后更新时间：2026-03-18

## 一、文档定位

本文件不再只描述“第二轮可以做什么”，而是作为三轮开发节奏中的中间总纲使用，目标是同时回答三个问题：

1. 当前系统已经做到哪里。
2. 最终系统还需要具备哪些能力。
3. 第二轮必须完成哪些工作，才能为第三轮收尾与答辩打好基础。

当前前提保持不变：

- 项目继续基于现有系统增量开发，不推翻重做。
- 认证状态继续使用 `sessionStorage`。
- 教师账号继续使用工号登录。
- Neo4j 不能成为论文录入和画像展示的强依赖，必须保留 MySQL 回退链路。
- 新增页面继续接入现有严格路由守卫。

## 二、当前系统开发进度

当前项目可视为：`V1.1-portrait-baseline` 主链路已形成，第一轮工程收口已完成，系统已经具备可演示、可继续扩展的基础版本。

### 当前已稳定可用的功能

- 认证与会话基础能力
  - 教师使用工号登录。
  - 前端使用 `sessionStorage` 保存登录态。
  - 页面已接入路由守卫。
- 教师基础信息管理
  - 支持教师注册、登录、资料查看和资料编辑。
  - 管理员具备教师信息维护基础能力。
- 科研成果主链路
  - 论文成果录入可用。
  - 已有 BibTeX 导入与导入前确认能力。
  - 其他成果类型已有基础录入结构。
- 教师画像展示
  - 已能展示教师核心画像数据。
  - 已有画像聚合与评分基础。
- 学术关系拓扑展示
  - 已有图谱展示页面。
  - Neo4j 不可用时仍可走回退链路。
- 工程收口基础
  - `type-check` 可通过。
  - `manage.py check` 可通过。
  - 关键后端模块已有最小必要测试。
  - 已形成接口清单、验收边界、演示步骤等文档基线。

### 当前已存在但还不够完整的功能

- 成果管理仍偏“录入导向”，编辑、筛选、搜索等能力不足。
- 会话管理仍偏“能登录”，续期与统一失效处理不足。
- 图谱仍偏“能展示”，同步收口与维护性不足。
- 管理端仍偏“能看能改基础信息”，缺少更稳的治理视角。
- 推荐、学院看板、RAG、复杂图分析等能力存在页面或预留方向，但不应表述为当前已完成。

## 三、系统最终目标

由于总开发轮次只规划三轮，因此第二轮不能只盯着修补主链路，还需要把最终系统目标提前写清楚，避免第三轮时缺少结构基础。

最终系统建议达到以下目标：

### 1. 主链路目标

系统能够稳定完成以下完整闭环：

1. 教师使用工号登录。
2. 教师维护个人基础信息。
3. 教师录入或导入论文及其他科研成果。
4. 系统基于真实成果生成并展示教师科研画像。
5. 系统展示教师学术关系拓扑。
6. 管理员能够查看教师数据并进行必要治理。

### 2. 管理与展示目标

- 教师画像支持更完整的指标展示与解释。
- 学院级看板支持按学院、年份等维度查看核心统计。
- 重要页面具备更稳定的异常处理、权限边界和演示路径。

### 3. 智能与扩展目标

最终系统可以包含但目前尚未完成的能力：

- 项目指南与推荐系统深化
- 学院级看板二期钻取与导出
- RAG / 智能问答
- 复杂图分析与推理
- 画像快照、离线聚合、推荐缓存等扩展表设计

注意：

- 上述能力是“最终系统目标的一部分”，不是“当前已完成能力”。
- 第二轮的任务不是把这些全部做完，而是为第三轮留下清晰、可接的结构基础。

## 四、第二轮的核心目标

第二轮开发的定位应当是：

`以主链路稳固为中心，以最终系统扩展为约束，完成第三轮开发所需的基础收口。`

换句话说，第二轮要同时完成两类事情：

- 第一类：把一期主链路补稳，避免第三轮还在修基础问题。
- 第二类：把最终系统会需要但现在尚未完成的能力，提前做好接口边界、数据边界、模块边界和文档边界。

## 五、第二轮必须完成的事项

以下内容建议作为第二轮真正要落地的工作，而不是停留在愿景层面。

### 任务 1：认证与会话收口

本轮应完成：

- 统一 401 处理。
- 明确 token 过期后的页面跳转、提示和恢复策略。
- 在不改 `sessionStorage` 的前提下，补齐会话续期或失效处理机制。

为什么本轮必须做：

- 所有页面都依赖这条链路。
- 如果第二轮不收口，第三轮新增任何高级功能都会反复踩认证问题。

是否纳入当前阶段验收：

- 是。

### 任务 2：成果管理链路增强

本轮应完成：

- 补齐成果编辑能力。
- 增加基础筛选、搜索或分类查看。
- 继续补该链路的类型约束、接口约束和最小测试或验证步骤。
- 让成果录入页不再只是“能新增”，而是“能维护、能演示”。

为什么本轮必须做：

- 画像和图谱都依赖成果数据质量。
- 如果成果链路维护能力不足，第三轮智能分析和推荐就没有稳定数据基础。

是否纳入当前阶段验收：

- 是。

### 任务 3：图谱链路统一收口

本轮应完成：

- 梳理图谱同步入口，减少重复同步或双路径冲突。
- 保持 Neo4j 可选、MySQL 回退可用。
- 明确图谱页面依赖的查询层和失败降级策略。
- 补图谱权限与回退场景的最小验证。

为什么本轮必须做：

- 图谱是演示亮点，也是高风险模块。
- 第三轮若要做复杂图分析，必须先有可控、稳定的基础图数据链路。

是否纳入当前阶段验收：

- 是。

### 任务 4：用户信息单源治理

本轮应完成：

- 明确 `CustomUser` 与教师档案数据的主从关系。
- 收拢同步逻辑，减少字段漂移。
- 补教师资料更新、管理员修改相关的回归测试。

为什么本轮必须做：

- 画像、推荐、管理端都会依赖教师基础信息。
- 如果数据源不稳，第三轮继续扩展只会增加维护成本。

是否纳入当前阶段验收：

- 是。

### 任务 5：前端类型与测试基线继续补齐

本轮应完成：

- 继续给成果页、管理页、推荐页、看板页补共享类型。
- 逐步减少页面内部散落的临时类型定义。
- 为关键页面逻辑或组合逻辑补最小前端测试，或者至少形成可复现验证脚本。

为什么本轮必须做：

- 第二轮结束时，如果没有更稳的类型和测试基线，第三轮很难安全扩展二期能力。

是否纳入当前阶段验收：

- 是。

## 六、第二轮需要兼顾但不要求完整做完的最终能力

下面这些能力虽然不要求在第二轮全部交付，但第二轮必须把它们纳入设计边界，不能等到第三轮再从零考虑。

### 1. 项目指南与推荐系统深化

第二轮至少要做到：

- 明确当前推荐能力的真实边界，不把预留能力写成已完成。
- 梳理推荐依赖的教师特征、成果特征和解释字段。
- 在文档里标清第三轮是继续规则增强，还是进入更强的推荐实现。

第二轮不要求：

- 不强行上复杂模型。
- 不把推荐作为当前一期主链路强验收内容。

第三轮承接目标：

- 做到推荐结果更可解释、更稳定，并能成为展示亮点之一。

### 2. 学院级看板二期钻取与导出

第二轮至少要做到：

- 确认当前学院看板使用的真实聚合口径。
- 明确哪些筛选、钻取、导出是第三轮准备做的。
- 避免第二轮改动破坏现有聚合接口。

第二轮不要求：

- 不必一次性完成完整钻取和导出。

第三轮承接目标：

- 支持更明确的学院、年份、教师维度切换，以及可演示的导出能力。

### 3. RAG / 智能问答

第二轮至少要做到：

- 明确其仍属于未完成方向。
- 如果代码中已有预留模块，应整理边界、避免误用。
- 在文档中给出未来接入所依赖的数据来源和前置条件。

第二轮不要求：

- 不在当前轮次强行接入完整问答功能。

第三轮承接目标：

- 能基于教师画像、成果或知识片段做基础问答演示。

### 4. 复杂图分析与推理

第二轮至少要做到：

- 先把基础图谱数据链路收稳。
- 预留复杂图分析所需的数据口径说明。
- 不让当前图谱实现和未来分析能力相互冲突。

第二轮不要求：

- 不直接做复杂路径分析、社区发现或高级推理成品。

第三轮承接目标：

- 在已有稳定图谱基础上，增加 1 到 2 个可展示的高级分析能力。

### 5. 画像快照、离线聚合、推荐缓存等扩展表设计

第二轮至少要做到：

- 不假设这些表已经存在。
- 梳理哪些能力未来确实需要扩展表支撑。
- 在文档中明确哪些可以继续走实时聚合，哪些更适合第三轮再引入扩展表。

第二轮不要求：

- 不急于新增一批未验证收益的表结构。

第三轮承接目标：

- 根据最终展示需求，按最小必要原则新增扩展表或缓存层。

## 七、第二轮结束时应该达到的状态

如果第二轮做得正确，结束时系统应达到以下状态：

- 主链路稳定性明显提升。
- 成果管理不再只有录入能力，还具备基本维护能力。
- 图谱链路不再是高耦合黑盒。
- 教师信息同步边界更加清晰。
- 前端类型和测试基线更完整。
- 推荐、看板、RAG、复杂图分析、扩展表设计等最终能力虽然未全部完成，但都已经有明确边界和第三轮承接条件。

## 八、第三轮的承接重点

第三轮不应再承担“大量修基础”的任务，而应基于第二轮的基础，集中完成最终系统的展示增强和能力补足。

第三轮建议重点：

1. 完成推荐系统的展示级增强。
2. 完成学院看板二期的核心钻取或导出能力。
3. 选择性落地一个轻量级 RAG / 问答演示链路。
4. 在稳定图谱上增加 1 到 2 个复杂图分析亮点。
5. 按需引入画像快照、离线聚合或缓存方案。
6. 做最终答辩演示路径优化与全链路验收。

## 九、建议的第二轮执行顺序

为了兼顾主链路稳定和第三轮准备，建议本轮按以下顺序推进：

1. 认证与会话收口
2. 成果管理链路增强
3. 图谱链路统一收口
4. 用户信息单源治理
5. 前端类型与测试基线继续补齐
6. 同步补推荐、看板、RAG、复杂图分析、扩展表设计的边界文档

## 十、结论

第二轮不是简单补 bug，也不是直接冲第三轮功能，而是整个三轮开发中最关键的“中台收口阶段”。

第二轮如果完成得好，第三轮就可以把精力放在：

- 强化展示效果
- 补足最终亮点能力
- 完成答辩导向的整体收尾

而不是继续反复修主链路和基础结构。
