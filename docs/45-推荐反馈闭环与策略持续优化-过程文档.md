# 45-推荐反馈闭环与策略持续优化-过程文档

## 1. 本次任务目标

本轮工作的目标不是重写推荐系统，而是在现有规则增强型推荐基础上，把“推荐给出结果”推进到“推荐可反馈、可回看、可汇总、可用于后续规则优化”的最小闭环。

本次坚持以下边界：

- 当前推荐仍是规则增强路线，不引入复杂机器学习推荐。
- 当前先落地轻量反馈闭环，再评估后续更强的排序优化。
- 不因为新增反馈而破坏原有推荐结果、收藏、画像联动、管理员对比和问答联动主链路。

## 3. 当前基线梳理

### 3.1 已有能力

仓库当前已具备以下反馈基础：

- 收藏持久化：`ProjectGuideFavorite`
- 推荐历史快照：`ProjectGuideRecommendationRecord`
- 轻量反馈信号：
  - `INTERESTED`
  - `NOT_RELEVANT`
  - `PLAN_TO_APPLY`
  - `APPLIED`
- 推荐页已可提交反馈、查看推荐历史、展示基础反馈分布
- 管理员页已有推荐分布摘要

### 3.2 主要断点

虽然基础设施已经存在，但闭环还不够完整：

- 教师反馈录入过于轻量，缺少备注承接，难以形成可复盘材料。
- 反馈摘要结构较弱，教师端和管理员端都不容易看出“响应情况如何”。
- 管理员摘要更偏推荐分布，还没有清晰地表达教师响应覆盖率、计划申报、已申报等结果。
- 接口虽已有历史记录，但“反馈影响排序”边界没有在数据结构和页面说明中表达清楚。

## 4. 本次实现内容

### 4.1 复用现有最小反馈记录结构

本次没有新增表，继续复用现有：

- `ProjectGuideFavorite`
- `ProjectGuideRecommendationRecord`

其中 `ProjectGuideRecommendationRecord` 作为当前最小反馈闭环载体，承接：

- 推荐结果快照
- 收藏快照
- 反馈信号
- 反馈备注
- 最后反馈时间

这符合“最小必要、可维护、可回退”的原则，也避免为了闭环新增冗余结构。

### 4.2 反馈摘要增强

后端 `feedback_summary` 现在新增并返回：

- `responded_guide_count`
- `current_recommendation_count`
- `response_rate`
- `favorite_count`
- `interested_count`
- `plan_to_apply_count`
- `applied_count`
- `not_relevant_count`
- `positive_feedback_count`
- `negative_feedback_count`
- `latest_feedback_at`
- `recent_feedback_items`
- `strategy_note`

说明：

- `distribution` 现在按“每个指南的最新反馈状态”汇总，避免一条指南历史多次反馈把当前状态放大失真。
- `feedback_record_count` 仍保留累计反馈记录数，用于表示长期积累规模。

### 4.3 教师反馈输入增强

推荐页反馈按钮改为“打开反馈记录弹窗”，支持：

- 继续使用原有 4 类轻量反馈信号
- 可选填写反馈备注

这让“感兴趣”“计划申报”“暂不相关”等信号不再只是一个按钮，而是能附带最小可复盘上下文。

### 4.4 教师端闭环展示增强

推荐页新增闭环展示信息：

- 反馈覆盖率
- 已反馈数量 / 当前推荐数量
- 收藏数
- 计划申报数
- 已申报数
- 最近反馈列表
- 反馈策略边界说明

这样教师端不再只是“点一下按钮”，而是能看到自己已经形成了怎样的反馈积累。

### 4.5 管理员响应分析增强

管理员摘要新增响应类指标：

- 反馈覆盖率
- 已反馈指南数
- 正向反馈数
- 暂不相关数
- 计划申报数
- 已申报数
- 累计反馈记录数

管理员分析卡片也同步改为更偏“推荐响应闭环”的表达，而不是只有推荐分布。

### 4.6 为后续排序优化预留边界

本次新增明确数据边界字段：

- `data_meta.feedback_ranking_boundary`

用途：

- 明确当前只是预留“反馈影响排序”的数据结构和解释边界
- 明确当前不会因为少量反馈直接重排推荐
- 为后续规则迭代或人工复盘提供稳定接口边界

## 5. 关键设计决策与权衡

### 5.1 不新增新表

原因：

- 现有 `ProjectGuideRecommendationRecord` 已能承载最小反馈闭环
- 新增表会增加迁移、兼容和维护成本
- 当前阶段更适合先把已有结构真正用起来

### 5.2 不把反馈直接写入推荐排序

原因：

- 当前反馈样本量可能很小
- 直接重排容易制造“看起来更智能，实际上不可解释”的假优化
- 先把反馈采集、摘要和历史打通，更符合当前阶段目标

### 5.3 管理员只看摘要，不代替教师反馈

原因：

- 反馈本质上应来自教师本人
- 管理员当前只做观察和复盘，不直接替教师提交反馈
- 保持教师 / 管理员轻量边界，不扩展成复杂 RBAC

## 7. 最小人工验证步骤

### 7.1 教师视角

1. 进入“项目指南推荐”页。
2. 选择一条推荐结果，点击：
   - 感兴趣
   - 计划申报
   - 已申报
   - 暂不相关
3. 在弹窗中填写或不填写备注后保存。
4. 检查页面是否同步更新：
   - 当前反馈标签
   - 最近反馈时间
   - 闭环概览
   - 最近反馈列表
   - 历史抽屉中的反馈记录

### 7.2 管理员视角

1. 以管理员进入推荐页。
2. 切换到某位已有反馈记录的教师。
3. 检查管理员推荐分析中是否出现：
   - 反馈覆盖率
   - 正向反馈数
   - 计划申报数
   - 已申报数
4. 确认管理员可查看教师响应摘要，但仍不能替教师触发收藏或反馈操作。

## 9. 当前边界与后续建议

当前已交付：

- 收藏闭环
- 轻量反馈闭环
- 反馈摘要
- 管理员响应分析
- 推荐历史回看

当前未做：

- 基于反馈自动重排推荐结果
- 复杂反馈质量建模
- 复杂多阶段推荐实验平台

后续建议：

- 当 `INTERESTED / PLAN_TO_APPLY / APPLIED / NOT_RELEVANT` 样本积累到足够规模后，可先做“规则系数人工调优”，而不是直接上学习排序。
- 若后续确实需要做反馈影响排序，建议仍以“可解释规则加权”作为过渡层，再评估是否进入更复杂模型化推荐。
