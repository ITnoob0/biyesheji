# 30-统一日志、异常提示与错误回退治理-过程文档

## 1. 本轮目标

在当前系统已经完成多轮功能增强的基础上，补一套最小可用的统一异常治理机制，重点解决以下问题：

- 接口异常时前端提示口径碎片化
- 后端异常可能直接暴露调试页或原始 HTML
- 登录、推荐、问答、图谱四条关键链路的降级说明不够统一
- 缺少系统级请求编号与排查抓手

本轮不追求完整日志平台建设，而是优先完成：

- 统一后端错误结构
- 统一前端错误解析与展示
- 关键链路统一回退提示
- 文档化的环境差异与排查顺序

## 2. 扫描结论

### 2.1 后端现状

扫描结果表明：

- `backend/core/settings.py` 中此前没有配置 DRF 全局 `EXCEPTION_HANDLER`
- 中间件链中此前没有系统级请求编号和 API 兜底异常中间件
- 多数接口依赖 DRF 默认异常结构
- 少量接口仍存在手写 `Response({"detail": ...}, status=...)` 分支

高频异常类型主要包括：

- 认证失败 / 账户停用
- 权限不足
- 参数校验失败
- 资源不存在
- 运行时异常导致的 500

### 2.2 前端现状

扫描结果表明：

- 已有共享请求层 `frontend/src/utils/http.ts`
- 已有部分错误文案工具 `frontend/src/utils/authPresentation.js`
- 但推荐、问答、图谱等页面仍各自手写 `ElMessage.error(...)`
- 登录链路虽然已拦截 HTML 调试页，但统一错误编号和下一步建议还没有收口

### 2.3 高风险链路

本轮优先治理：

- 登录
- 项目推荐
- 智能问答
- 学术图谱

原因：

- 这些链路最常被演示和联调使用
- 一旦异常，用户最容易直接感知到系统是否“可理解、可恢复”

## 3. 本轮改动

### 3.1 后端统一错误结构

新增：

- `backend/core/api_errors.py`

作用：

- 统一生成标准错误结构
- 将 DRF 默认异常包装为统一格式
- 保留 `detail`，避免前端旧逻辑立即失效
- 补充 `error.code`、`error.next_step`、`request_id`

当前统一错误结构核心字段：

- `detail`
- `request_id`
- `error.code`
- `error.message`
- `error.status`
- `error.recoverable`
- `error.next_step`

### 3.2 请求编号与 API 兜底异常中间件

新增：

- `backend/core/middleware.py`

包括：

- `RequestIdMiddleware`
- `ApiExceptionMiddleware`

效果：

- 每个请求都会得到统一 `request_id`
- API 未处理异常会被转换为统一 JSON 错误结构
- 后端控制台日志会记录带 `request_id` 的异常信息

### 3.3 DRF 全局异常处理接入

更新：

- `backend/core/settings.py`

新增：

- `REST_FRAMEWORK["EXCEPTION_HANDLER"] = "core.api_errors.custom_exception_handler"`

并把：

- `DJANGO_DEBUG`

改为环境变量控制，默认仍兼容当前开发环境。

### 3.4 手写错误分支收口

本轮同步收口了几个高频手写分支：

- `backend/users/views.py`
  - 教师账户不存在
- `backend/achievements/views.py`
  - 教师不存在
  - BibTeX 预览格式错误
- `backend/graph_engine/views.py`
  - 学术图谱无权访问提示改为统一中文口径
- `backend/project_guides/views.py`
  - 推荐参数异常提示明确为“推荐参数校验未通过”

### 3.5 前端统一错误解析层

新增：

- `frontend/src/utils/apiFeedback.js`
- `frontend/src/utils/apiFeedback.d.ts`

作用：

- 统一解析后端标准错误结构
- 统一提取：
  - 错误消息
  - 下一步建议
  - 请求编号
- 继续过滤 HTML 调试页和 Traceback 文本，避免原样展示

### 3.6 关键链路前端回退提示统一

更新：

- `frontend/src/views/LoginView.vue`
- `frontend/src/views/ProjectRecommendationView.vue`
- `frontend/src/views/AssistantDemoView.vue`
- `frontend/src/views/AcademicGraph.vue`
- `frontend/src/utils/http.ts`
- `frontend/src/utils/authPresentation.js`

关键效果：

- 登录失败不再只弹一条分散消息，而是带统一错误提示区
- 推荐失败会给出“仍可继续使用其他主链路”的提示
- 问答异常时页面级降级说明与结构化 fallback 结果保持一致
- 图谱异常时页面级提示与图谱回退说明更统一
- 401 失效跳转时，统一使用标准错误消息而不是固定死文案

## 4. 开发环境与正式环境差异

### 4.1 开发环境

- `DJANGO_DEBUG=1`
- 后端控制台仍保留完整堆栈，便于本地排查
- API 响应优先返回统一 JSON 错误结构，不再把调试页直接暴露给前端页面

### 4.2 正式环境

- 应设置 `DJANGO_DEBUG=0`
- 用户不应直接看到 Django 调试页或后端堆栈
- 页面只展示统一错误消息、下一步建议和请求编号

注意：

- 当前本轮没有建设完整日志平台
- 当前“统一日志”主要指请求编号、后端控制台日志口径和错误排查抓手统一

## 5. 系统级错误排查说明

推荐排查顺序：

1. 记录前端提示中的 `request_id`
2. 执行 `python backend/scripts/startup_preflight.py`
3. 确认迁移已执行
4. 执行 `python backend/scripts/run_validation_pipeline.py`
5. 根据 `request_id` 查看后端控制台或服务日志
6. 如果是图谱问题：
   - 确认 Neo4j 是否启用
   - 确认 MySQL 回退链路仍可用
7. 如果是问答或推荐问题：
   - 确认目标教师、指南或学院筛选参数是否存在

## 6. 历史遗留问题与本轮边界

### 6.1 历史遗留问题

- 过去部分页面是各自定义错误提示
- 过去后端缺少统一 `request_id`
- 过去 API 500 场景可能让前端收到 HTML 调试页

### 6.2 本轮未做

- 没有引入外部日志平台
- 没有接入 ELK / Loki / Sentry 等系统
- 没有统一改造全部业务接口成功响应结构
- 没有对所有页面都补齐页面级异常面板

本轮强调的是最小闭环，而不是一次性做完整观测平台。

重点关注：
