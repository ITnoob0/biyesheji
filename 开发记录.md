# 开发设想

要把这个系统做扎实，体现出独立思考的深度，我们就必须跳出传统“教务处科研管理后台”（单纯的增删改查）的思维定式。一个真正有价值的系统，其核心逻辑应该是：**用数据理解人，用算法服务人。**

我们可以按照从“底层数据流转”到“顶层业务价值”的逻辑，将系统最终要实现的功能划分为四大核心模块。

### 一、 数据治理与特征提取模块（底座）

这是最容易被忽视，但却是决定系统成败的“脏活累活”。没有干净、准确的数据，后面的智能辅助都是空中楼阁。

- **多源异构数据接入：** * 不能只靠教师手动填表。系统需要支持从知网、Web of Science 等公开文献库导入格式化数据（如 BibTeX），同时支持解析教师上传的 PDF 版结题报告或专利文件，自动提取关键信息。（优先做教师主动上传功能，系统爬取和导入在系统开发近乎完成以后再进行）
- **学者同名消歧（痛点功能）：** * 中文名重名率极高。系统需要通过“合作者网络”、“所属机构”和“研究关键词”进行交叉比对，解决“李雷的论文被错误归属到另一个李雷名下”的问题。
- **动态标签生成：**
  - 使用 NLP 技术对教师近五年的论文摘要进行主题词抽取。不依赖教师自己设定的静态研究方向，而是通过真实产出生成动态的“技术栈标签”和“研究热点标签”。

### 二、 多维立体画像模块（核心）

画像不仅仅是基础数据的罗列，而是响应国家“破五唯”导向，对教师科研状态与综合价值的量化诊断。

- **科研生命周期轨迹图（纵向追踪）：** - 融合时间轴技术，追踪教师的研究焦点变迁。例如，清晰呈现一位学者如何从“经典机器学习”平滑过渡到“多模态大模型微调”，以动态视角反映其学术敏锐度与技术迭代能力。
- **全息综合评价雷达模型（横向诊断）：** - 打破传统的“唯论文论”，构建六维评估体系：
    1. **基础学术产出：** 高水平期刊/会议论文数量、被引频次、H-Index。
    2. **经费吸附与攻坚：** 纵向基金项目（国自然/省部级）与横向企业合作课题（金额与数量）。
    3. **硬核知识产权：** 授权发明专利、软件著作权登记及实际成果转化情况。
    4. **人才培养成效：** 指导学生参与高水平学科竞赛（如挑战杯、大创项目）、教改项目、指导优秀硕博论文。
    5. **学术活跃与声誉：** 期刊编委/审稿人任职、国际会议特邀报告、学术组织任职。
    6. **跨学科融合度：** 论文与项目的跨学科交叉指数（如AI+医疗、AI+数字人文等）。
- **学术社交拓扑图（关系网络）：** - 利用图数据库可视化该教师的合作网络。不仅挖掘校内外的显性合作者，更通过共同的研究关键词识别潜在的学术圈子与隐形“学术梯队”，支持机构层面的团队建设。

### 三、 智能辅助与主动推荐模块（高光）

这是体现“智能”二字的业务层，要把系统从“静态档案馆”变成“科研小助手”。

- **项目/基金精准“相亲”：**
  - 系统定期抓取国家级、省部级科研项目申报指南。通过语义相似度计算，将指南中的“攻关需求”与教师画像中的“技术积累”进行匹配。主动向教师推送：“该省重点研发计划与您近两年的研究契合度达85%，建议关注”。
- **跨学科“学术红娘”：**
  - 当教师发起一个大型交叉学科项目时（例如：AI+医疗），系统能基于全院/全校的画像库，推荐具有互补技能的潜在合作者，打破学科壁垒。
- **自动生成学者主页/简历：**
  - 支持一键导出排版精美的个人学术简历或网页端学者主页，免去教师在不同场合（评职称、报项目、对外交流）反复排版填表的繁琐工作。

### 四、 宏观决策看板模块（面向管理层）

不仅服务于单体教师，还要为学院的资源配置提供上帝视角。

- **科研梯队健康度诊断：** * 分析学院内各年龄段、各职称的科研产出分布。识别出“断层”风险（比如某方向严重依赖一位老教授，缺乏青年接班人）。
- **高潜人才挖掘：** * 通过算法识别出近期科研产出呈现指数级上升趋势，但目前职称或资源获取量较低的“黑马”教师，提示学院给予重点资源倾斜。

------

**下一步的思考方向：**

这四个模块构成了系统的全貌。对于前期的开发而言，贪大求全容易烂尾。

如果要兼顾技术可行性和项目的出彩程度，我建议**一期工程先死磕“画像模块”和“智能辅助模块中的‘项目精准推荐’”**。

你倾向于我们先深入讨论哪一个功能的具体实现逻辑？比如，如何设计数据库表结构来支撑“学术社交拓扑图”，或者如何设计匹配算法来实现“项目推荐”？





# 技术栈

这是一份为您精简整理的技术栈清单，仅包含技术名称及其在项目中的具体应用位置：

### 一、 前端与可视化

- **Vue.js 3 + TypeScript**：前端核心框架与开发语言，用于构建整个系统的用户交互界面。
- **Element-Plus**：UI 组件库，用于快速搭建后台管理页面的表格、表单、导航栏等基础组件。
- **Apache ECharts**：基础数据可视化引擎，用于渲染教师的能力雷达图、科研产出趋势折线图等常规统计图表。
- **D3.js**：高级图形渲染库，专门用于绘制和交互高复杂度的“学者社交网络 / 合作者关系拓扑图”。

### 二、 后端核心

- **Python 3.11+**：后端核心编程语言。
- **Django 4.x**：后端核心 Web 框架，负责处理基础业务逻辑、数据路由与 ORM 映射。
- **Django REST Framework (DRF)**：API 开发框架，用于向前端提供标准化的 RESTful JSON 接口。
- **Simple JWT**：安全认证模块，用于实现前后端分离架构下的用户登录鉴权与会话保持。

### 三、 数据库与缓存

- **MySQL 8.0+**：核心关系型数据库，用于存储教师基本信息、论文/项目元数据结构化数据，并通过其 JSON 字段存储动态画像标签。
- **Neo4j**：核心图数据库，专门用于存储“人与人”、“人与项目”之间的复杂节点与边，支持高效的图谱关系查询。
- **Redis**：内存数据库，用作高频查询数据的缓存层，同时作为后台异步任务的消息代理（Broker）。

### 四、 AI 与智能辅助

- **Ollama**：大模型本地运行环境，用于在本地服务器私有化部署开源大语言模型（如 Qwen 或 GLM），确保科研数据隐私。
- **LangChain**：大模型应用编排框架，用于将本地大模型与数据库连接，实现 RAG（检索增强生成），处理项目指南匹配和文本生成。
- **Pandas + Scikit-learn**：数据分析与机器学习库，用于前期异构数据的清洗、预处理，以及基础的学者特征聚类算法。

### 五、 异步与辅助工具

- **Celery**：分布式任务队列，用于在后台静默处理耗时操作（如：大模型长文本推理、图谱节点批量更新）。
- **Requests + BeautifulSoup4**：轻量级爬虫组件，用于定期自动化抓取外部科研网站的公开项目指南和学术动态。
