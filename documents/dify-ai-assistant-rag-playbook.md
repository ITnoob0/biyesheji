# Dify AI 助手 RAG 配置与知识库增强手册

更新时间：2026-05-03

## 1. 推荐建设方案

在 Dify 中建立一个专门面向本项目的知识库：

```text
BiShe-科研评价与智能问答核心知识库
```

这个知识库不是通用百科库，而是服务以下项目场景：

1. 科研评价规则解释。
2. 政策来源追溯。
3. 真实演示成果认定说明。
4. 教师画像、成果结构、项目推荐、学院看板问答。
5. Dify 接入链路、页面上下文和角色差异说明。

## 2. 文档导入顺序

建议按以下顺序导入：

1. `documents/research-evaluation-core-rules-final.md`
2. `documents/research-evaluation-source.txt`
3. `documents/rule-achievement-real-demo-sources.md`
4. `documents/dify-ai-assistant-project-context.md`
5. `documents/dify-knowledge-base-upload-manifest.md`

前三个是必须文档，后两个是项目上下文增强文档。

## 3. 分段策略

### 3.1 核心规则文件

文件：`research-evaluation-core-rules-final.md`

建议：

- 按标题层级切分。
- 保留表格完整性。
- 不要把一个积分表切碎到无法保留“项目类别 + 积分”的程度。
- 对“最终不纳入系统的内容”“通用认定规则”“成果录入边界”保持较高召回优先级。

推荐检索重点：

- 成果是否录入。
- 是否参与总分。
- 是否进入能力雷达。
- 积分规则。
- 剔除项。
- 就高原则。

### 3.2 政策来源文件

文件：`research-evaluation-source.txt`

建议：

- 按章节、附件、表格块切分。
- 允许保留较长父段，因为用户会问“政策依据是什么”。
- 对文件号、发文日期、总则、认定规则、激励程序等内容保持关键词可命中。

推荐检索重点：

- 延大发〔2022〕52 号。
- 高质量贡献评价与激励标准（科研部分 2022 版）。
- 学术贡献、社会贡献、定性评价。
- 同一成果就高原则。
- 学术不端、知识产权、争议认定。

### 3.3 真实演示来源文件

文件：`rule-achievement-real-demo-sources.md`

建议：

- 每位教师/每条成果单独成为一个强语义块。
- 保留规则条目、成果名称、公开来源、采用事实、时间归一说明。
- 规则编号如 `PROJECT_NS_06`、`PAPER_HS_07`、`AWARD_HS_05`、`TRANS_09` 必须可被关键词检索命中。

推荐检索重点：

- 演示成果为什么这样认定。
- 公开来源是什么。
- 是否保守映射。
- 时间为什么这样归一。

### 3.4 项目上下文文件

文件：`dify-ai-assistant-project-context.md`

建议：

- 按接入链路、输入变量、页面归一化、典型回答策略切分。
- 保留 JSON 示例，便于 Dify 理解输入变量。

推荐检索重点：

- Dify Base URL。
- `/chat-messages`。
- `/workflows/run`。
- `role`、`department`、`title`、`context_hint`、`route_path`。
- `current_page` 页面归一化。
- 回退策略。

## 4. Dify 应用类型建议

### 4.1 首选：Chat App + 知识库

项目后端优先请求 `/chat-messages`，因此推荐使用 Dify Chat App。Chat App 应绑定上述知识库，并开启知识检索。

优势：

- 支持 `conversation_id` 多轮会话。
- 与当前前端悬浮助手天然匹配。
- 后端已能读取 Dify 返回的 `answer` 和 `conversation_id`。

### 4.2 可选：Workflow App

如果你已经使用 Codex 生成了 Dify YAML/DSL 工作流，也可以使用 Workflow App。后端已预留 `/workflows/run` 回退调用。

Workflow 的输入变量应与后端一致：

| 变量 | 含义 |
| --- | --- |
| `draft` | 用户问题 |
| `current_role` | 当前角色：`admin` 或 `teacher` |
| `current_page` | 归一化页面 |
| `page_context` | 页面上下文 |
| `answer_style` | 回答风格 |
| `language` | 语言 |

Workflow 的 End 节点建议输出变量名设置为以下任一项：

```text
answer
output
text
result
```

原因：后端会按这些字段依次提取工作流答案。

## 5. 推荐检索参数

不同 Dify 版本配置项名称可能略有差异，下面给出项目推荐值：

| 项 | 建议值 | 说明 |
| --- | --- | --- |
| 索引方式 | 高质量 | 规则表格与政策文本需要更稳检索 |
| 检索方式 | 混合检索优先 | 兼顾规则编号、关键词和语义问法 |
| Top K | 5 到 8 | 规则解释通常需要多条依据 |
| Score Threshold | 0.35 到 0.55 | 过高可能漏掉政策来源，过低会引入噪声 |
| Rerank | 开启，如可用 | 长规则文档建议开启重排 |
| 分段长度 | 700 到 1200 中文字符 | 兼顾表格完整性和召回精度 |
| 分段重叠 | 80 到 150 中文字符 | 避免表格标题和内容断裂 |

## 6. 推荐回答模板

### 6.1 成果认定类

```text
结论：该成果在当前系统中【可录入/不录入/需补充材料后判断】。

依据：
- 命中的规则条目：……
- 规则说明：……
- 如涉及政策来源：……

解释：
……

下一步建议：
请补充/核验……

边界：最终认定仍以学校科研管理部门或学术委员会审核为准。
```

### 6.2 演示成果来源类

```text
结论：该演示成果按【规则条目】认定。

采用事实：
- 成果名称：……
- 来源页面：……
- 关键事实：……

映射理由：
……

边界：这是演示数据映射口径，不替代正式业务审核。
```

### 6.3 教师画像建议类

```text
结论：基于当前系统数据，教师画像呈现……

依据：
- 成果结构：……
- 画像/推荐/图谱上下文：……

建议：
1. ……
2. ……
3. ……

边界：画像是系统数据下的辅助解释，不是对教师学术水平的绝对评价。
```

## 7. 建议 Prompt 片段

可以放入 Dify Chat App 的系统提示词：

```text
你必须优先使用知识库检索结果回答。涉及科研评价规则时，先依据最终规则文件判断系统口径；涉及政策出处时再引用政策来源文件；涉及演示成果时引用真实演示来源清单；涉及系统接入方式时引用项目上下文文件。

当知识库未检索到依据时，不得编造规则、分值、政策名称或演示来源。请明确回答“当前知识库未覆盖该问题”，并提示需要补充哪些材料。

回答应使用简体中文，风格简洁执行型。优先结构：结论、依据、解释、下一步建议、边界说明。
```

## 8. 上线前验收

### 8.1 Dify 控制台检索测试

逐条测试：

1. `核心科研能力评价最终保留哪些成果？`
2. `平台与团队为什么不进入核心科研能力规则？`
3. `国家自然科学基金地区科学基金项目是多少分？`
4. `延大发〔2022〕52号文件主要规定了什么？`
5. `曹新亮演示成果为什么是 PROJECT_NS_06？`
6. `周学军专利转让为什么是 14.4 分？`

### 8.2 项目前端联调测试

1. 设置后端环境变量：

```bash
set DIFY_BASE_URL=https://api.dify.ai/v1
set DIFY_API_KEY=app-xxxxxxxxxxxxxxxx
set DIFY_REQUEST_TIMEOUT=30
```

2. 启动后端：

```bash
cd backend
python manage.py runserver
```

3. 启动前端：

```bash
cd frontend
cmd /c npm run dev
```

4. 使用教师或管理员账号登录。
5. 打开悬浮 AI 助手。
6. 测试以下问题：

```text
请解释当前系统为什么只保留核心科研能力规则。
请结合当前页面说明我可以问哪些问题。
如果某成果属于科普项目，系统是否录入？
我在项目推荐页时，助手应该如何解释推荐理由？
```

## 9. 常见问题处理

### 9.1 Dify 返回空答案

检查：

1. Dify App 是否发布。
2. API Key 是否来自当前 App。
3. App 是否绑定知识库。
4. End 节点输出变量是否为 `answer`、`output`、`text` 或 `result`。

### 9.2 项目前端提示回退模式

检查后端环境变量：

```text
DIFY_BASE_URL
DIFY_API_KEY
DIFY_REQUEST_TIMEOUT
```

同时确认 `DIFY_BASE_URL` 不要写成完整的 `/chat-messages` 地址，只写 API 根地址。

### 9.3 回答没有引用核心规则

检查 Dify 知识库：

1. `research-evaluation-core-rules-final.md` 是否已上传并索引完成。
2. 检索参数是否 Top K 太低。
3. Score Threshold 是否过高。
4. 分段是否把表格标题和内容拆散。

### 9.4 回答把剔除项当成可录入

需要在系统提示词中强化：

```text
凡命中最终规则文件中的“最终不纳入系统的内容”和“直接不录”部分，应优先按剔除项处理，不得因政策来源原文中出现该项而自动认定为当前系统可录入。
```

## 10. 与项目代码的关系

本手册不要求改动数据库表，也不要求新增后端向量检索服务。当前设计是：

```text
前端悬浮助手 -> Django Dify 代理接口 -> Dify Chat/Workflow App -> Dify 知识库 RAG -> 返回答案
```

项目本地仍保留原有安全回退和规则说明能力。Dify 是主要 AI 能力来源，但项目后端负责统一鉴权、页面上下文传递、异常回退和前端响应适配。
