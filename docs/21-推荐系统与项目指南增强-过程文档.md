# 推荐系统与项目指南增强过程文档

## 1. 任务目标与本次边界

本次工作在现有“项目指南管理 + 规则型推荐结果页 + 推荐理由展示”基础上做增量增强，目标是把推荐模块升级为更易解释、可筛选对比、便于后续持续增强的推荐能力。

本次明确保持以下边界：

- 不改工号登录主流程。
- 不改 `sessionStorage` 登录态方案。
- 不引入复杂机器学习推荐模型。
- 不推翻现有项目指南模型、推荐接口和推荐页面。
- 不把“未来模型化推荐”描述成已上线能力。

## 2. 仓库扫描与现状梳理

本次首先扫描了以下推荐相关模块：

- 后端：
  - `backend/project_guides/models.py`
  - `backend/project_guides/serializers.py`
  - `backend/project_guides/services.py`
  - `backend/project_guides/views.py`
  - `backend/project_guides/tests.py`
- 前端：
  - `frontend/src/views/ProjectRecommendationView.vue`
  - `frontend/src/views/ProjectGuideManagementView.vue`
  - `frontend/src/views/project-guides/recommendationHelpers.js`
  - `frontend/src/views/project-guides/types.ts`
  - `frontend/src/types/projectGuides.ts`

扫描结果确认：

- 当前推荐核心来自 `ProjectGuideRecommendationService`，属于规则增强型推荐。
- 当前推荐依据主要包括：研究方向、研究兴趣、论文关键词、学科/院系、近三年成果活跃度、申报窗口等。
- 当前管理员已经可以指定教师查看推荐，但缺少明确的对比能力和管理视角分析结果。
- 前端已有基础搜索、分类筛选和排序，但标签体系、解释维度、对比展示和收藏标记仍较弱。

## 3. 本次增量设计

### 3.1 规则配置增强

在现有项目指南模型上增加两类最小必要配置字段：

- `rule_profile`
  - 用于表达该条指南更偏向哪种规则策略。
  - 当前支持：`BALANCED`、`KEYWORD_FIRST`、`DISCIPLINE_FIRST`、`WINDOW_FIRST`、`ACTIVITY_FIRST`。
- `recommendation_tags`
  - 用于表达管理员给指南补充的业务标签。
  - 示例：`教育场景`、`重点指南`、`适合青年教师`。

这两类字段均直接复用当前指南管理入口维护，不新增独立配置后台。

### 3.2 推荐解释增强

推荐结果在原有“推荐理由”基础上，新增：

- `recommendation_labels`
  - 面向教师和管理员展示的推荐标签。
  - 例如：`高匹配`、`主题贴合`、`方向贴合`、`近期可申报`、`适合活跃教师`。
- `explanation_dimensions`
  - 面向解释层的结构化维度说明。
  - 当前覆盖：
    - 主题标签匹配
    - 学科方向匹配
    - 活跃度支撑
    - 申报窗口友好度
    - 资助信息明确度
    - 规则配置加成

这部分仍是规则计算结果，不是模型推理解释。

### 3.3 教师推荐对比

管理员视角下，推荐接口新增 `compare_user_id` 参数，用于对两位教师的同一批指南做规则得分对比。

当前对比能力包括：

- 对比教师画像摘要
- 每条指南的 `compare_score`
- 当前教师与对比教师的 `compare_delta`
- 简短 `comparison_summary`
- 汇总级 `comparison_summary`
  - 当前教师更优数量
  - 对比教师更优数量
  - 持平数量
  - 差异最大的指南标题

权限边界：

- 仅管理员允许使用 `compare_user_id`
- 教师本人不可调用指定教师对比能力

### 3.4 管理员分析视角增强

推荐接口在管理员视角下补充 `admin_analysis`，用于支持推荐结果的管理分析展示。

当前包含：

- 推荐总量
- 关注等级分布
- 规则档位分布
- 高频推荐标签

本次没有新增管理员独立推荐分析页面，而是复用既有推荐页，通过管理员身份在同一受保护入口展示增强信息。

### 3.5 推荐交互增强

推荐结果页增强了以下交互：

- 指定教师推荐
- 管理员双教师对比
- 按指南级别筛选
- 按关注等级筛选
- 按推荐标签筛选
- 按对比分差排序
- 本地收藏标记

其中“收藏”采用浏览器 `sessionStorage` 本地标记：

- 仅作为当前登录态下的本地辅助能力
- 不写入后端数据库
- 不宣称为跨端同步收藏体系

## 4. 当前推荐能力口径说明

### 4.1 当前路线

当前推荐系统仍属于“规则增强路线”，不是机器学习或深度模型推荐系统。

当前推荐结果主要用于：

- 帮助教师快速理解为什么某条指南值得关注
- 帮助管理员从规则与标签层面审阅推荐结果
- 帮助后续迭代积累更稳定的标签、规则和反馈数据

### 4.2 什么条件下才考虑进入更强模型化推荐

只有在以下条件较稳定时，才建议评估更强模型化推荐：

- 规则特征体系已较稳定，频繁调整明显减少
- 项目指南文本解析和标签抽取口径稳定
- 教师侧已积累足够的点击、收藏、申请反馈等行为样本
- 管理员对推荐结果的人工校准数据具备持续积累价值

在这些条件不满足前，继续沿用规则增强路线更可控、更容易解释。

## 5. 当前能力边界与未实现项

本次未实现以下能力，并明确不对外描述为已上线：

- 复杂机器学习推荐模型
- 语义召回 + 重排序全链路
- 基于大规模行为数据的协同过滤
- 跨端同步收藏体系
- 推荐反馈闭环训练
- 自动学习型规则调参后台

## 7. 最小人工验证步骤

建议按以下主链路人工验证：

1. 管理员进入“项目指南管理”页面，新建或编辑一条指南。
2. 为指南配置规则档位和推荐标签。
3. 管理员进入“项目推荐”页面，切换指定教师查看推荐结果。
4. 继续选择第二位教师，观察推荐对比区、对比分差和汇总摘要。
5. 在推荐结果页尝试按级别、关注等级、推荐标签和收藏状态筛选。
6. 检查推荐卡片中是否能看到：
   - 推荐理由
   - 推荐标签
   - 解释维度
   - 规则档位
   - 收藏按钮
7. 检查管理员分析区是否能展示推荐总量、分布信息和高频标签。

这属于本地运行环境问题，不属于本次推荐模块代码逻辑新增问题。
